Техническое Задание (ТЗ): Telegram Web App Магазин (Nuxt 3)
1. Описание проекта
Гибридное Telegram-приложение (Bot + Web App Mini App) для заказа товаров/услуг. Приложение работает без классической базы данных (Stateless). Вся коммуникация, управление статусами заказов и уведомления строятся на базе Telegram Bot API (вебхуки и inline-кнопки).

2. Технологический стек
Фреймворк: Nuxt 3 (Vue 3, Composition API, <script setup>).
Язык: TypeScript.
Стилизация: Tailwind CSS (для быстрого UI).
Стейт-менеджер: Pinia (для корзины на фронтенде).
Telegram API: Нативный fetch (на бэкенде Nuxt Nitro) + скрипт Telegram Web App на фронтенде.
3. Роли и Функционал
3.1. Клиент (Покупатель)
Чат с ботом:
Команда /start -> Бот присылает приветствие и Inline-кнопку "Открыть магазин" (Web App).
Команда /help -> Бот присылает контакты поддержки.
Web App (Фронтенд):
Стилизация адаптируется под тему Telegram (светлая/темная) через window.Telegram.WebApp.
Список товаров (Mock-данные: Картинка, Название, Цена).
Возможность добавить товар в корзину.
Отображение нативной кнопки Telegram внизу экрана (MainButton) при наличии товаров в корзине (текст: "Оформить заказ на X руб.").
При нажатии на MainButton -> отправка данных заказа на бэкенд Nuxt, показ уведомления об успехе, закрытие Web App (WebApp.close()).
Уведомления: Получает текстовые сообщения от бота при смене статуса заказа менеджером.
3.2. Менеджер
При создании заказа клиентом, менеджер (указанный в .env по CHAT_ID) получает сообщение.
Формат сообщения менеджеру:
Номер заказа, Список товаров, Итоговая сумма, Username клиента.
Inline-клавиатура с кнопками статусов.
Флоу статусов (Inline-кнопки):
Новый заказ -> Кнопка [Принять в работу].
Нажал "Принять" -> Сообщение обновляется, кнопка меняется на [Передать курьеру]. Клиенту летит уведомление.
Нажал "Передать курьеру" -> Кнопка меняется на [Доставлен]. Клиенту летит уведомление.
Нажал "Доставлен" -> Кнопки пропадают, остается текст "Заказ завершен". Клиенту летит уведомление.
4. Архитектура и Передача данных (Без БД)
Данные между менеджером и клиентом связываются через скрытый параметр callback_data в кнопках Telegram (лимит 64 байта).
Формат callback_data: status_userId_orderId (например: work_123456789_8812).
status (work, courier, done) - какое действие совершил менеджер.
userId - Telegram ID клиента (чтобы знать, кому слать уведомление).
orderId - случайный ID заказа (для красоты).

6. Логика серверных маршрутов (Nitro)
server/api/order.post.ts
Принимает JSON от Web App (корзина, данные юзера из initData).
Генерирует короткий номер заказа.
Формирует сообщение.
Отправляет запрос к Telegram API (sendMessage) в чат менеджера с прикрепленной inline-клавиатурой (в callback_data зашит ID клиента).
Возвращает фронтенду 200 OK.
server/api/webhook.post.ts
Принимает обновления от Telegram Bot API.
Если это message (текст):
Проверяет, если текст /start, отправляет сообщение с кнопкой Web App.
Если это callback_query (нажатие кнопки менеджером):
Парсит callback_data (разделяет по _).
В зависимости от статуса отправляет sendMessage клиенту (по userId из даты).
Обновляет сообщение менеджера через editMessageText (меняет текст кнопок на следующие по логике).
Отправляет answerCallbackQuery, чтобы у менеджера пропали "часики" загрузки на кнопке.
