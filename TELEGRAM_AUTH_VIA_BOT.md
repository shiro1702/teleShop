## Авторизация «через Telegram‑бота»: какие есть варианты

Этот файл отвечает на вопрос: **можно ли авторизовывать пользователя «через Telegram‑бота» вместо стандартного Login Widget, и какие вообще есть варианты для веба и Mini App.**

---

### 1. Что уже сделано в проекте

Сейчас в проекте используются два разных механизма:

- **Mini App / Telegram WebApp** — авторизация через `initData`:
  - В Mini App при запуске Telegram передаёт в `window.Telegram.WebApp.initData` подписанную строку с данными пользователя.
  - Фронт отправляет `initData` на бэкенд (в нашем случае в `/api/order`).
  - Бэкенд валидирует подпись по официальной схеме (HMAC‑SHA256 с `bot_token`) и по результату считает, что «этот запрос пришёл от конкретного Telegram‑юзера».
- **Внешний сайт (Web)** — авторизация через **Telegram Login Widget**:
  - В шапке сайта есть кнопка `TelegramLoginButton.client.vue`, которая вставляет официальный виджет `https://telegram.org/js/telegram-widget.js?22`.
  - После успешного логина фронт шлёт данные в `/api/auth/telegram`, бэкенд проверяет `hash`, создаёт сессию и кладёт httpOnly‑cookie `tg_session`.
  - Все дальнейшие запросы с сайта (например, `/api/order`) отождествляют пользователя через эту сессию.

Оба варианта уже являются авторизацией **«через бота»**, просто в разных контекстах:

- внутри Telegram (Mini App + `initData`);
- снаружи (веб‑сайт + Login Widget / Web Login).

---

### 2. Варианты авторизации «через бота» в экосистеме Telegram

Условно есть три официальных направления:

1. **Telegram Mini Apps (WebApp + `initData`)**
   - Подход **для приложений внутри Telegram**.
   - Telegram сам аутентифицирует пользователя, а Mini App получает подписанные данные.
   - Бэкенд валидирует `initData` по ключу `bot_token` и строит свои сессии (JWT / cookie).
   - В проекте этот вариант уже используется для TMA.

2. **Log In With Telegram для внешних сайтов**
   - Подход **для обычных веб‑сайтов**.
   - Бот выступает «идентичностью» приложения, но веб‑логин делается через:
     - либо **Telegram Login Widget** (устаревший, но широко используемый),
     - либо более новый **Telegram Login Library / OpenID Connect**.
   - Механика:
     - пользователь жмёт кнопку на сайте;
     - Telegram показывает подтверждение (какой бот, какой домен);
     - после согласия на сайт прилетают данные (`id`, `username`, `auth_date`, `hash`);
     - сервер валидирует `hash` (HMAC‑SHA256 с SHA256(bot_token));
     - создаётся своя сессия/токен.
   - Это именно тот путь, который сейчас реализован через `/api/auth/telegram`.

3. **URL Authorization / `keyboardButtonUrlAuth` (логин из чата бота)**
   - Вариант, когда **всё начинается внутри диалога с ботом**:
     - в сообщении бота есть кнопка с `login_url` / `keyboardButtonUrlAuth`;
     - при клике Telegram показывает пользователю подтверждение «бот X хочет авторизовать вас на домене Y»;
     - после согласия клиент открывает URL с параметрами авторизации и подписью;
     - бэкенд опять-таки проверяет подпись и создаёт сессию.
   - Это тоже авторизация «через бота», но инициируется из чата, а не с сайта.

Все три механизма используют **один и тот же базовый принцип**:

> Telegram выдаёт подписанный набор данных (через виджет, URL auth или `initData`), а ваш сервер на своей стороне **криптографически проверяет подпись** и только после этого доверяет ID пользователя.

---

### 3. Есть ли «магический» способ авторизации _только_ через бота, без Login Widget

Если под этим понимать что‑то в духе:

> «Пусть пользователь просто напишет боту `/start`, а сайт по user_id поверит, что это он же»

— то **официально такой вариант не считается безопасным**:

- нет подписи, привязанной к конкретному HTTP‑запросу на сайт;
- нельзя надёжно связать сессию в браузере и факт общения с ботом (кроме самодельных схем с одноразовыми кодами/токенами);
- Telegram в своих доках явно продвигает **Login (widget / OIDC) и URL auth** как каноничные способы.

То есть реально варианты такие:

- **В Mini App** — только через `initData` (что вы уже делаете).
- **На внешнем сайте**:
  - использовать официальный **Login Widget / Log In With Telegram** (как у вас),
  - либо URL‑авторизацию из чата (`keyboardButtonUrlAuth`),
  - либо новый OIDC‑вариант (тот же «Log In With Telegram», но через стандартный OpenID Connect‑клиент).

Просто «юзер написал боту → мы поверили его ID на сайте без подписи» — это всегда самописный и потенциально уязвимый костыль.

---

### 4. Какой вариант логично использовать именно в этом проекте

Для вашей схемы «омниканал: сайт + Mini App» сейчас выбрана и, по сути, оптимальная связка:

- **Mini App**:
  - auth через `initData` → `/api/order` (TMA‑режим).
- **Веб‑сайт**:
  - кнопка «Войти через Telegram» (Login Widget),
  - `/api/auth/telegram` валидирует `hash` и поднимает cookie‑сессию,
  - `/api/order` использует эту сессию, если нет `initData`.

Альтернативно, если захочется «чистого via bot» для входа на сайт:

- можно добавить пользователю кнопку в боте с `login_url` (`keyboardButtonUrlAuth`);
- человек кликает её **в чате с ботом**, попадает на сайт уже с авторизационными параметрами в URL;
- сервер проверяет подпись и поднимает ту же `tg_session`.

Но это всё равно **официальный URL auth / Log In With Telegram**, а не какая‑то отдельная «механика только через бота».

### 5. Новая схема: привязка Telegram‑аккаунта к аккаунту на сайте через одноразовые токены

Цель этой схемы — **отвязать авторизацию на веб‑сайте от старого Login Widget** и вместо этого:

- хранить основную аутентификацию и сессии в Supabase (`auth.users`);
- использовать Telegram‑бота **только для привязки/идентификации** Telegram‑аккаунта к уже существующему пользователю сайта;
- сделать механизм универсальным, чтобы потом его можно было переиспользовать и для других клиентов (desktop / mobile / внешние интеграции).

Ключевая идея:

> Бот создаёт **одноразовый токен** и привязывает его к `telegram_id`. Пользователь по ссылке с этим токеном попадает на сайт, авторизуется (через Supabase), и уже сервер на своей стороне **без Login Widget** аккуратно привязывает `telegram_id` к его профилю.

#### 5.1. Структура данных в Supabase

Используются две таблицы:

- **`profiles`** — расширение стандартной таблицы `auth.users`:
  - `id uuid` — PK, `references auth.users(id) on delete cascade`;
  - `telegram_id bigint unique` — Telegram‑ID пользователя (один Telegram‑аккаунт не может быть привязан к двум профилям);
  - `created_at timestamptz default now()`;
  - `updated_at timestamptz default now()` + триггер на автообновление.
- **`auth_tokens`** — одноразовые токены для привязки:
  - `token uuid primary key default gen_random_uuid()`;
  - `telegram_id bigint not null`;
  - `created_at timestamptz default now()`;
  - `expires_at timestamptz default (now() + interval '15 minutes')`.

RLS‑политики:

- для `profiles` — стандартное правило: **пользователь видит/меняет только свою запись** (`auth.uid() = id`);
- для `auth_tokens` — **полный доступ только у `service_role`**:
  - обычные клиенты (браузер, Mini App) не могут создать/прочитать/обновить/удалить токены;
  - все операции с токенами идут через серверный ключ Supabase (или Nitro‑обёртку `serverSupabaseServiceRole`).

Это делает токены чисто серверной сущностью и не даёт фронтенду «играться» с чужими Telegram‑ID.

#### 5.2. Поведение Telegram‑бота (`/login`)

Логика при команде `/login` у бота:

1. Бот получает `telegram_id` пользователя из `ctx.from.id`.
2. Генерирует `uuid` (одноразовый токен).
3. Через `supabase-js` (с **service_role**‑ключом) вставляет запись в `auth_tokens`:
   - `token = uuid`,
   - `telegram_id = ctx.from.id`,
   - `expires_at` по умолчанию = `now() + 15 минут`.
4. Отправляет пользователю сообщение с inline‑кнопкой:
   - текст, например: «Привязать Telegram к аккаунту на сайте»;
   - URL вида:  
     `https://<site-domain>/link-telegram?token=<UUID>`.

Пользователь нажимает кнопку, Telegram открывает браузер с этой ссылкой — дальше всё делает веб‑часть.

#### 5.3. Серверный API‑роут Nitro: `/api/auth/link-telegram` (POST)

На стороне Nuxt/Nitro создаётся эндпоинт `server/api/auth/link-telegram.post.ts` со следующей ответственностью:

1. Принять `token` из `body`.
2. Проверить, что пользователь **авторизован в Supabase**:
   - через `serverSupabaseUser(event)` получить `user`;
   - если `user == null` — вернуть `401 Unauthorized`.
3. Получить сервисный клиент Supabase:
   - `const serviceClient = await serverSupabaseServiceRole(event);`
4. Найти токен в таблице `auth_tokens`:
   - `select * from auth_tokens where token = :token limit 1`;
   - если не найден — ошибка `400 Invalid token`.
5. Проверить срок действия:
   - сравнить `expires_at` с текущим временем;
   - если токен просрочен — удалить его и вернуть `400 Token expired`.
6. Взять `telegram_id` из строки `auth_tokens` и выполнить upsert в `profiles`:
   - `upsert({ id: user.id, telegram_id }, onConflict: 'id')`;
   - при необходимости можно дополнительно проверить, что этот `telegram_id` ещё не занят другим `id`.
7. Удалить использованный токен из `auth_tokens` (даже если upsert уже прошёл успешно).
8. Вернуть JSON с `success: true` и, например, `telegramId` для дебага.

Таким образом, Telegram‑ID никогда не приходит с фронта «как есть» — он всегда идёт через **серверный слой** + одноразовый токен, созданный ботом.

#### 5.4. Веб‑страница Nuxt: `/link-telegram`

На фронтенде создаётся страница `pages/link-telegram.vue` (Composition API, `<script setup>`), которая работает по следующему сценарию:

1. При монтировании читает `token` из query‑параметров: `?token=<UUID>`.
2. Проверяет текущего пользователя через `useSupabaseUser()`:
   - **если пользователь не авторизован**:
     - показывает текст «Для привязки Telegram необходимо войти на сайт»;
     - кнопка «Войти» перекидывает на страницу логина (`/login`) с `redirect=/link-telegram` и тем же `token` в query, чтобы после логина вернуться обратно.
   - **если пользователь уже авторизован**:
     - показывает кнопку «Привязать Telegram».
3. При нажатии на «Привязать Telegram»:
   - делает `POST /api/auth/link-telegram` с `body: { token }`;
   - обрабатывает состояния:
     - `isLoading` — дизейбл кнопки и отображение прогресса;
     - `isSuccess` — сообщение «Telegram‑аккаунт успешно привязан»;
     - `errorMessage` — показ текста ошибки в случае `4xx/5xx`.

4. После успешной привязки выполняется переход в каноничную tenant-корзину:
   - `/:city_slug/:tenant_slug/cart` (резолв по `shop_id`);
   - legacy `/checkout?shop_id=...` не используется.

Страница не делает никаких прямых запросов к `auth_tokens`/`profiles` — всё общение с базой идёт только через серверный API и сервисный ключ.

#### 5.5. Безопасность и универсальность подхода

Почему это безопаснее и универсальнее, чем хранить собственную «телеграм‑сессию»?

- **Разделение ответственности**:
  - Supabase управляет аутентификацией (e‑mail/пароль, OAuth и т.д.);
  - Telegram‑бот только выдаёт привязанные к `telegram_id` одноразовые токены.
- **Минимизация доверия к фронту**:
  - браузер никогда не может сам «подставить» произвольный `telegram_id`;
  - единственный источник `telegram_id` — запись в `auth_tokens`, которую создаёт бот по факту общения в Telegram.
- **Одноразовость и TTL**:
  - токен живёт ограниченное время (15 минут);
  - после использования удаляется;
  - даже если ссылка утечёт, её можно использовать только один раз и только пока не истёк TTL.
- **Расширяемость для других клиентов**:
  - в будущем можно добавлять другие площадки (desktop / native‑клиенты), которые будут открывать ту же `/link-telegram`‑страницу в браузере или вызывать её через встроенный web‑view;
  - серверный контракт (`auth_tokens` + `/api/auth/link-telegram`) остаётся тем же — меняется только «откуда» пользователь переходит по ссылке.

### 6. Предложения по дальнейшим шагам

Чтобы довести схему до production‑уровня и сделать её максимально удобной и безопасной, имеет смысл:

1. **Усилить контроль уникальности привязки**
   - Перед `upsert` в `profiles` дополнительно проверять, что `telegram_id` не привязан к другому `id`.
   - При попытке привязать уже занятый `telegram_id` — отдавать человеку понятное сообщение в интерфейсе и/или предлагать отвязку.

2. **Добавить механику отвязки Telegram**
   - На странице профиля пользователя реализовать кнопку «Отвязать Telegram»:
     - `update profiles set telegram_id = null where id = auth.uid()`;
   - Для критичных сценариев — просить повторное подтверждение (пароль / e‑mail‑код и т.п.).

3. **Фоновая очистка просроченных токенов**
   - Настроить в Supabase `cron`/Edge Function или внешний воркер, который регулярно удаляет записи из `auth_tokens` с `expires_at < now()`.
   - Это не критично для безопасности, но уменьшает «мусор» в таблице.

4. **Аудит и логирование операций привязки**
   - Ввести отдельную таблицу `telegram_link_logs` или использовать существующий логгер:
     - `user_id`, `telegram_id`, `action` (`link`/`unlink`), `ip`, `user_agent`, `created_at`.
   - Это упростит расследование инцидентов и отладку.

5. **UX‑улучшения на фронтенде**
   - После успешной привязки:
     - показывать понятный экран с подтверждением и краткой инструкцией, что теперь может делать бот (уведомления, быстрые действия из чата и т.д.);
     - предлагать сразу перейти в профиль или вернуться в нужное место (например, в корзину/оформление заказа).

6. **Интеграция с дальнейшими каналами (desktop / mobile / другие платформы)**
   - Выделить `/link-telegram` и `/api/auth/link-telegram` как **часть публичного API** проекта:
     - задокументировать формат URL и тела запроса/ответа;
     - договориться, что любые будущие клиенты, которым нужно связать Telegram с аккаунтом пользователя, следуют именно этому протоколу.
   - При необходимости добавить аналогичные одноразовые токены для других платформ (`platform_tokens`), используя ту же архитектурную идею.

7. **Постепенный отказ от старого подхода**
   - Поэтапно вынести/отключить старую авторизацию через Telegram Login Widget для веб‑версии:
     - временно оставить её только как fallback (для части пользователей);
     - после успешной обкатки нового механизма и Telegram‑привязки — полностью удалить устаревший код и документацию по старому способу, оставив только актуальную схему.

Если хочешь, можно вынести разделы 5–6 в отдельный файл (`TELEGRAM_LINK_FLOW.md`) и оформить это как формальную спецификацию протокола привязки Telegram‑аккаунта к профилю пользователя.
