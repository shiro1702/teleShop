- Валидация состава корзины по доступности товаров в городе и ресторанах бренда.

## Состав строки корзины: параметры и модификаторы

Клиент должен однозначно видеть, **что именно** он заказал, без открытия модалки товара.

### Параметры позиции (вариант товара)

- Показывать **человекочитаемую подпись** выбранного варианта: например «30 см», «350 мл», «8 шт».
- Если у варианта заданы **атрибуты** (вес, объём, количество), выводить их рядом: «30 см · 590 г», «350 мл», «8 шт».
- Для вторичных выборов **без изменения цены** (например «тонкое / традиционное тесто» при одинаковой цене) — тоже показывать в строке, чтобы не путать состав порции.
- **Цена строки** в корзине = цена выбранного варианта параметра ± эффект модификаторов (доплаты и множители). Пользователю достаточно видеть **итог за единицу** и **итог по количеству**; детализацию «база + модификаторы» можно свернуть в подпись при необходимости.

### Модификаторы

- Список выбранных опций (названия), **без дублирования мелких доплат в бейджах**, если итоговая цена строки уже финальная (как на текущей витрине).
- Для множителей и нестандартных правил в заказе/Telegram допускается явная пометка (например `×1.1`).

### Поведение при изменении количества

- Увеличение количества той же строки сохраняет **тот же** набор параметров и модификаторов (отдельная строка — только если отличается конфигурация).

## Апсейл и кросс-сейл в корзине (План)

Для увеличения среднего чека в корзине должен быть реализован блок рекомендаций.

### 1. Добор до цели (Goal-based)
- **Логика:** Если до бесплатной доставки (или применения промокода) не хватает суммы `X`, показываем блок: *"До бесплатной доставки осталось X ₽"*.
- **Выдача:** Карусель товаров, цена которых чуть больше или равна `X` (чтобы одним кликом закрыть цель). Идеально подходят напитки, соусы, десерты.

### 2. Кросс-сейл (С этим часто берут)
- **Логика:** Анализ текущего состава корзины.
- **Ручная настройка:** В админке к товару или категории привязываются "Рекомендуемые товары" (например, к пицце — соусы, к бургеру — картошка).
- **Автоматическая (fallback):** Если в корзине нет напитков — вывести карусель категории "Напитки" с заголовком *"Не забудьте напитки"*.

### 3. UX добавления из апсейла
- Товары из блока рекомендаций должны добавляться в корзину **в один клик** (если у них нет обязательных параметров/модификаторов).
- Если параметры есть — открывается стандартная модалка настройки товара.

### Связь с документацией меню

- Модель параметров, примеры (пицца/морс/наггетсы) и схема данных — `pages/dashboard/menu.md`.
- Мультивыбор модификаторов с лимитом — `pages/dashboard/menu/modifiers.md`.

## Проверки при отправке корзины / создании заказа *(TODO для сервера и при необходимости клиента)*

Перед финальным созданием заказа бэкенд **пересчитывает и валидирует** каждую строку. Клиент может дублировать часть проверок для UX (подсветка конфликтов), но **источник правды — сервер**. Ниже — чеклист по пунктам; реализация в `server/api/order.post.ts` (и связанных хелперах) должна закрывать каждый подходящий пункт.

### Стоп-лист и видимость

- [ ] Позиция не в **`is_in_stop_list`** для выбранного `restaurant_id` (`restaurant_product_overrides`).
- [ ] Позиция не **`is_hidden`** для этого филиала.
- [ ] Позиция **активна глобально** (`products.is_active`), категория активна, если это влияет на выдачу.
- [ ] *(когда будет схема)* Ни один **ингредиент** строки не в **`restaurant_ingredient_stops`** для этого филиала — см. `pages/dashboard/menu/ingredients.md`.

### Цена

- [ ] Итоговая цена строки **сверена с сервером**: база (с учётом `price_override` филиала и выбранного **варианта параметра**), затем модификаторы по правилам магазина; допустимый либо **строгий ноль расхождения** с телом запроса, либо явная политика округления, зафиксированная в коде.
- [ ] При расхождении — ответ с кодом конфликта цен и **актуальными** данными для обновления корзины на клиенте.

### Параметры позиции

- [ ] Переданный **вариант параметра** существует, **активен**, принадлежит этому `product_id`.
- [ ] Вариант **не отключён** оверрайдом на филиале (если введены филиальные оверрайды опций).
- [ ] **Цена варианта** на сервере совпадает с использованной при расчёте (или пересчитана сервером заново).

### Модификаторы

- [ ] Каждая выбранная опция принадлежит **разрешённой для товара** группе (товар + категория), группа и опция **активны**.
- [ ] Соблюдены **required / min_select / max_select** для каждой группы.
- [ ] Опции не из списка **отключённых** (`product_modifier_group_overrides` / `disabled_option_ids`).
- [ ] Доплаты и множители соответствуют актуальным данным в БД.

### Комбо *(когда появится сущность комбо в меню)*

- [ ] Состав комбо (шаги, выбранные позиции) соответствует **схеме комбо** и ограничениям шагов.
- [ ] Все вложенные позиции проходят те же проверки стопа, скрытия, цены, параметров и модификаторов.
- [ ] Цена комбо (фикс / сумма с правилами) совпадает с серверным пересчётом.

### Канал и способ получения *(см. `pages/dashboard/menu/channels.md`)*

- [ ] Выбранный способ получения (**доставка / самовывоз / зал / …**) **разрешён** настройками организации/филиала.
- [ ] *(уровень 2 каналов)* Позиция **разрешена** для текущего канала, если введена видимость по каналу.

### Прочее (связь с существующим планом)

- [ ] Филиал в **рабочем интервале** (см. блок ниже про график).
- [ ] Минимальная сумма заказа, зона доставки, оплата — по отдельным правилам checkout.

Детальное UX стопа на карточке товара (серая карточка vs скрытие) — `pages/dashboard/branches/[id]/menu.md` (раздел «UX витрины»).
