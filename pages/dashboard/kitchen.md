# Экран повара (KDS - Kitchen Display System)

Раздел для кухни: `/dashboard/branches/:id/kitchen` (или отдельное приложение для планшетов на кухне).

## Проблема текущего отображения заказов
Стандартный список заказов (Order 1: Пицца, Суши; Order 2: Пицца, Суп) неудобен для поваров при высокой нагрузке. Повару горячего цеха не нужно видеть суши, а пиццамейкеру удобнее закинуть в печь сразу 3 одинаковых пиццы из разных заказов, чем готовить их последовательно.

## Варианты реализации (Стратегия)

### Вариант 1: Сводный Kanban по позициям (Item-centric)
Вместо заказов на экране отображаются **карточки блюд**, которые нужно приготовить прямо сейчас, сгруппированные по названию.
- **Пример:** Блок "Картофель фри" — бейдж "x5 порций".
- **Действие:** Повар нажимает на карточку "Картофель фри", выбирает "Готово 3 порции". Система сама распределяет эти 3 порции в самые старые ожидающие заказы.
- **Плюсы:** Максимальная оптимизация работы кухни (batch cooking).
- **Минусы:** Сложнее реализовать логику частичной готовности заказа для сборщика.

### Вариант 2: Маршрутизация по цехам (Station-centric)
Заказ разбивается на подзаказы в зависимости от категории блюд.
- **Настройка:** В админке категории меню привязываются к "Цехам" (Горячий цех, Суши-бар, Пицца).
- **Отображение:** На планшете в горячем цеху появляются только карточки с супами и воками. В карточке указан номер заказа и таймер.
- **Сборка:** Отдельный экран "Сборщик" (Экспедитор), который видит весь заказ. Когда все цеха нажали "Готово" на своих частях, заказ у сборщика загорается зеленым — можно паковать.

### Вариант 3: Заказной вид с подсветкой дублей (Order-centric + Highlights)
Компромиссный и самый простой для MVP вариант.
- На экране классическая сетка заказов (карточки заказов).
- Но если в 3-х активных заказах есть "Маргарита", это блюдо внутри карточек подсвечивается одинаковым цветом или иконкой "x3 в очереди".
- Повар видит заказ целиком, но визуально быстро считывает, что можно объединить в готовке.
