# Экран повара (KDS - Kitchen Display System)

Раздел для кухни: `/dashboard/branches/:id/kitchen` (в планшетном режиме или отдельном фронтенде).

## Цель
Сделать повару вид, который оптимизирует кухню под **пачечную/параллельную готовку** и помогает быстро понять:
- какие позиции уже “на кухне” и в каком статусе
- какие блюда можно готовить параллельно (batch cooking)

## Контекст: несколько режимов работы и филиалы
У ресторана один и тот же бренд/меню может работать в разных **режимах получения** (channels), например:
- `delivery`
- `pickup`
- `dine-in`
- `qr-menu`
- `showcase-order`

На кухне важно разграничивать канбан:
1. **по филиалу**: доска строится только для выбранного `:id` (то есть `shop_id`)
2. **по режиму работы/каналу**: доска строится только для выбранного `fulfillment_type`

## Роли и доступ
Экран кухни показываем **всегда**, но:
- если у пользователя нет нужного доступа/роли для кухни — действия на UI **disabled** (read-only)
- подсветка дублей (MVP) может быть доступна в read-only режиме, даже если “готовка” еще недоступна

## Форматы канбана (табы)
На кухне используются табы:

### Таб 1 (Enabled в MVP): Подсветка дублей
Идея: показываем повару привычный “заказной” вид, но внутри карточек заказа **подсвечиваем позиции-дубли**.

Как определяется “дубликат” в MVP:
- берём активные `order_items` со статусами `pending`, `cooking`, `ready`
- группируем по `order_item`-идентификатору блюда (ключ: например `menu_item_id` / “позиция меню” в конкретном заказе)
- рассматриваем только в пределах текущего `shop_id` и выбранного `fulfillment_type`
- если одно и то же блюдо встречается в нескольких активных заказах — подсвечиваем и показываем бейдж вида `xN in queue`

UX-эффект:
- повар визуально считывает, что “Маргарита” есть в нескольких заказах
- он понимает, что можно готовить параллельно (хотя в MVP фактическое batch-распределение может быть недоступно)

### Таб 2 (Always visible, disabled в MVP): Batch cooking (full)
Планируем “полный batch” канбан (item-centric подход), где повар подтверждает готовность сразу **N порций**, а система распределяет их по самым старым ожидающим позициям.

В рамках MVP этот таб:
- отображается всегда для понимания будущей логики
- находится в disabled-состоянии (без действий)

## Стратегия реализации (выбор вариантов)
Исторически рассматривались варианты:
- Вариант 1: item-centric batch cooking (сложный state-менеджмент)
- Вариант 2: station-centric (по цехам)
- Вариант 3: order-centric + highlights (самый простой путь к MVP)

Для старта в этой дорожной карте выбирается:
- Enabled: Вариант 3 “Заказной вид с подсветкой дублей”
- Next (disabled сейчас): Вариант 1 “Сводный Kanban по позициям” (полный batch)

## План работ

### MVP (подсветка дублей)
1. Сделать эндпоинт получения активных позиций кухни:
   - источник: `order_items`
   - фильтры: `shop_id`, `fulfillment_type`
   - статусы: `pending`, `cooking`, `ready`
2. На фронтенде построить карточки заказов/позиций и подсветку дублей:
   - бейдж `xN` и одинаковый цвет/стиль для одной и той же позиции меню
3. Добавить таб “Batch cooking” с disabled состоянием (заглушка интерфейса).
4. Разрешить обновление данных без перезагрузки (планируется):
   - WebSocket/Realtime (позже/Next, зависит от готовности бэкенда).

### Next (полный batch cooking)
1. Реализовать “confirm batch”:
   - повар выбирает блюдо
   - указывает “готово N порций”
2. Система распределяет N порций по самым старым ожидающим позициям:
   - преимущество в очереди: по времени создания/очереди (pending по старшинству)
   - только в пределах текущего `shop_id` и `fulfillment_type`
3. Внести защиту от конкуренции:
   - атомарное распределение (транзакция/локи), чтобы два устройства не “забрали” одну и ту же порцию
4. Добавить журнал изменений/историю для трассировки ошибок.

### Later
- WebSocket/Realtime подписка на изменения `order_items` и заказов
- звуковые уведомления при падении новой активной позиции на кухню
- WIP-лимиты, SLA-индикаторы “проблемных” позиций
- отдельный “сборщик/экспедитор” экран (если выбирается station-centric дальше)

## Минимальные требования к API (для MVP/Next)
MVP:
- `GET /api/dashboard/branches/:id/kitchen?fulfillment_type=...`:
  - возвращает активные `order_items` + связанные данные для отображения (order id, позиция меню/label, qty, status, createdAt)

Next:
- `POST /api/dashboard/branches/:id/kitchen/batch-complete`:
  - входные параметры: `menu_item_id` + `qty`
  - сервер распределяет qty по pending-позициям и обновляет статусы
