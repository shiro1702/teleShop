## Платежи (эквайринг ресторана, B2C)

- Провайдеры для РФ: YooKassa и Т-Банк.
- В MVP ключи эквайринга хранятся на уровне `shop`.
- Владелец (`Owner`) указывает:
  - провайдера,
  - merchant/shop id,
  - secret key (в UI только маска).
- Платеж клиента создается только через серверный маршрут; клиент получает redirect на hosted payment page.
- Финальный статус подтверждается webhook от провайдера (не return URL).

### Что видит владелец

- Статус интеграции: `active` / `error` / `disabled`.
- Последняя успешная проверка и последняя ошибка.
- Маскированные реквизиты и кнопки: подключить, переподключить, отключить.

### Расширение на филиалы

- Базовая модель: merchant на уровне `shop`.
- Для сетей с разными договорами допускается переход на `restaurant` override с fallback на `shop`.

## Платежи платформы (подписка, B2B)

- SaaS-подписка оплачивается через отдельные merchant-ключи платформы.
- Поток подписки: checkout -> redirect -> webhook -> обновление `subscription status`.
- Поддерживаемые состояния: `active`, `past_due`, `canceled` (+ grace period по регламенту).
- Критичные операции (`смена плана`, `отмена подписки`) доступны только `Owner`.
