# Меню и каналы: доставка, самовывоз, зал, QR, витрина в телефоне *(план)*

Ресторан работает в разных **контекстах**: доставка, самовывоз, заказ в зале, меню по QR, «витрина» на телефоне у стойки. Вопрос — **одно ли меню и одни ли правила** на все каналы или нужны различия.

## Текущие зацепки в продукте

- В организации задаются **способы получения** (`delivery`, `pickup`, `dine-in`) и сценарий «витрина + к столу» (`to-table` / `pickup-point`) — см. `pages/dashboard/settings/organization.md`.
- У филиала в схеме есть флаги: `supports_dine_in`, `supports_qr_menu`, `supports_showcase_order` — см. миграцию `010_restaurants_fulfillment_modes.sql`.

Канал влияет на **UX оформления** (адрес, стол, оплата). Отдельный вопрос — нужно ли **разное меню/ассортимент** по каналу.

## Варианты модели (от простого к сложному)

### Уровень 1 — одно меню на бренд, канал только влияет на доставку *(рекомендация для MVP)*

- Каталог **одинаковый**; недоступность только через стоп/скрытие филиала и расписание.
- **Плюсы:** простая поддержка, один источник правды в дашборде, меньше ошибок.
- **Минусы:** нельзя убрать, например, «сложные комбо» только с доставки без обходных путей.

### Уровень 2 — видимость позиции/категории по каналу

- На товаре или категории: список **разрешённых каналов** (`delivery`, `pickup`, `dine_in`, `qr_menu`, `showcase`) или маска видимости.
- API витрины и заказа фильтруют выдачу по **текущему каналу** сессии (как пользователь зашёл: ссылка QR, приложение доставки и т.д.).
- **Стоп-лист филиала** накладывается поверх: если позиция в стопе — не заказать ни в одном канале (если не введён отдельный «стоп только для доставки» — сознательно не усложнять в MVP).

### Уровень 3 — отдельные привязки меню («снимки») на канал

- Несколько **представлений** одного глобального каталога: разный порядок, отключённые категории, отдельные цены.
- Дорого в поддержке и в UI дашборда; имеет смысл только при жёстких требованиях сети.

## Рекомендованная дорожная карта

1. **MVP:** уровень 1 + явно документировать, что QR/витрина/телефон используют **то же меню**, что и доставка/самовывоз, для выбранного филиала.
2. **Следующий шаг:** уровень 2 для 1–2 полей (`available_channels` на `products` или связь `product_channel_availability`), если появятся кейсы «в зале не продаём X».
3. Уровень 3 — только по запросу бизнеса.

## UX и витрина

- При входе по **QR** в параметрах URL или сессии фиксируется `restaurant_id` и режим `qr_menu`; каталог уже может быть сужен на уровне 2.
- **Витрина на телефоне** (`showcase_order`): тот же UI каталога, но упрощённый чекаут (без лишних шагов) — меню не обязано отличаться, если нет уровня 2.
- В дашборде филиала: блок **«Каналы»** (read-only в MVP) — какие режимы включены; редактирование видимости по каналу — когда появится уровень 2.
- Статусы доступности на витрине должны быть едиными между каналами (на уровне 1): `is_hidden` скрывает карточку, а `is_in_stop_list` / стоп по ингредиентам показываются как “временно нет” (см. `pages/dashboard/branches/[id]/menu.md`).
- Если включён уровень 2 (разная видимость по каналу): сначала фильтруем позиции/категории по разрешённым каналам, затем поверх применяем стоп-листы филиала (стоп не должен “обходиться” только потому, что пользователь в другом канале).
