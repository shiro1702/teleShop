# Ингредиенты товара *(план)*

Справочник **ингредиентов** на уровне организации (`shop_id`) и привязка их к товарам. Нужен для состава на витрине, фильтров («без арахиса», аллергены), массового **стопа по ингредиенту** на филиале и (опционально) отчётности.

## Зачем это не модификаторы

| | Ингредиенты | Модификаторы |
|---|-------------|--------------|
| Смысл | **Фактический состав** / этикетка / аллергены | **Выбор клиента** при заказе |
| Цена | Не влияют на цену сами по себе | Доплаты и множители |
| Стоп на филиале | Один переключатель → все товары с этим ингредиентом | Обычно отдельные опции или ручной стоп позиций |
| Пример | Моцарелла, арахис, глютен | «Доп. сыр», «Без лука» |

Ингредиент может совпадать по названию с опцией модификатора (например сыр), но в БД это **разные сущности**: модификатор описывает UX заказа, ингредиент — состав и правила фильтрации/стопа.

## Модель данных (целевая)

На уровне `shop_id`:

- **`ingredients`**
  - `id`, `shop_id`, `name` (отображаемое имя), `slug` или стабильный `code` для URL/аналитики (опционально)
  - `sort_order`, `is_active`
  - опционально: **`kind`** или теги — `allergen`, `vegan`, `spicy` и т.д. (для быстрых фильтров и бейджей; можно начать с простого списка без тегов)
  - опционально: `external_id` для кухни/POS

- **`product_ingredients`** (связь many-to-many)
  - `product_id`, `ingredient_id`
  - опционально: `quantity_note` (текст «по вкусу»), флаг **`may_contain`** («может содержать следы») — для юридически аккуратного отображения
  - `sort_order` — порядок строк в составе на витрине

Уникальность: `(product_id, ingredient_id)`.

### Стоп по ингредиенту на филиале

- **`restaurant_ingredient_stops`** (или поля в обобщённой таблице стопов)
  - `shop_id`, `restaurant_id`, `ingredient_id`
  - `is_stopped boolean` (или только факт строки = стоп)
  - опционально: `note`, `stopped_until` (автоснятие по времени — фаза 2)

**Эффективная доступность товара на филиале** (логика для API витрины и заказа):

1. Товар **скрыт** или в **стопе по позиции** → недоступен (см. раздел стопа по товарам в `pages/dashboard/branches/[id]/menu.md`).
2. Иначе: если у товара есть хотя бы один ингредиент из **`restaurant_ingredient_stops`** для этого филиала → товар считается **в стопе по составу** (то же поведение, что «в стоп-листе»: не заказать, но политика отображения — см. ниже).

Так не нужно дублировать N строк в `restaurant_product_overrides` при закончившемся одном ингредиенте; при необходимости кэш можно **денормализовать** фоновой задачей, но источник правды — стоп по ингредиенту + матрица `product_ingredients`.

### Приоритеты и конфликты

- Явный **стоп по товару** на филиале имеет смысл оставить независимым: менеджер может снять позицию вручную, даже если ингредиент ещё в стопе (редкий кейс) — **либо** запретить это правилом «ингредиент всегда сильнее». Для MVP зафиксировать одно правило в коде и в UI подсказкой:
  - **Рекомендация:** `стоп по товару OR стоп по любому ингредиенту состава` → позиция недоступна; отдельного «пробить» без снятия ингредиента нет (проще для кухни).

## Витрина (клиент)

### Отображение в карточке и модалке товара

- Блок **«Состав»** (или «Ингредиенты»): список имён в порядке `sort_order`.
- Позиции с `may_contain` — отдельная мелкая строка «Может содержать: …».
- Опционально: иконки/бейджи по тегам (аллергены, острое).

### Отображение в корзине и заказе
- В **корзине** достаточно компактного блока состава (например “Ингредиенты: …” с возможностью раскрыть при необходимости), чтобы гость видел состав без открытия модалки.
- В **заказе** и **Telegram** сохранять снимок состава на момент создания заказа (как минимум: список ингредиентов и информация `may_contain`), чтобы изменения меню позже не ломали юридически важный контент.
- Политика объёма текста:
  - если ингредиентов много, в корзине показывать первые N (например 5-7) и “ещё …” с раскрытием;
  - в модалке товара показывать полный список.

### Фильтры каталога

- **Исключающий фильтр по ингредиентам** (основной сценарий для аллергий): пользователь отмечает «без арахиса», «без молока» → показываются товары, у которых **нет** этих ингредиентов в `product_ingredients`.
  - Опционально (настраиваемо): исключать также товары, у которых ингредиент только в флаге `may_contain`.
  - Если выбрано несколько ингредиентов в исключающем режиме: логика `AND NOT` (товар не должен содержать ни одного выбранного).
- **Включающий фильтр по ингредиенту** (ключевой сценарий “ставить TSOP на все товары с ингредиентом”): пользователь выбирает ингредиент → показываются товары, в составе которых есть этот ингредиент.
  - Если выбрано несколько ингредиентов во включающем режиме: логика `OR` (товар содержит хотя бы один выбранный) или `AND` (содержит все) — зафиксировать в настройках/в выборе UI.
- **Теги/группы ингредиентов** (фаза 2 или MVP-лайт): если у ингредиента есть `kind`/теги, фильтры могут быть по группе (например “аллергены”, “веган”, “острое”), но источник данных остаётся `product_ingredients`.
- Фильтры должны работать в связке с **филиалом** и **стопом**:
  - товары в стопе по позиции (`restaurant_product_overrides.is_in_stop_list`) и/или в стопе по ингредиенту (`restaurant_ingredient_stops`) считаются недоступными;
  - в каталог может попадать карточка в приглушённом виде (“временно нет”), если выбрана политика отображения стопов для витрины (см. UX в `pages/dashboard/branches/[id]/menu.md`).

### Стоп по ингредиенту на витрине

- Все товары, содержащие остановленный на филиале ингредиент, ведут себя как **в стоп-листе** для этого ресторана (бейдж, блокировка «В корзину»), без ручного переключения каждой позиции.

## Дашборд: глобальное меню

### `/dashboard/menu/ingredients` *(новый раздел)*

- Таблица справочника: название, активность, порядок, (опционально) `kind`/теги (аллерген, веган, острое) и признак “стандартный/нестандартный” для UI.
- Создание / редактирование / архивация (`is_active = false` вместо удаления, если уже привязан к товарам).
- Поиск и дубликаты: при вводе имени подсказки «похожие» (Fuzzy/search by normalized name) чтобы не плодить «Моцарелла» и «Моцарелла ».
- Жизненный цикл:
  - `is_active=false` означает “не показывать в форме выбора ингредиентов на товарах”;
  - при редактировании товара уже привязанные (даже неактивные) ингредиенты остаются, но помечаются как “неактивен” (чтобы менеджер мог решить: переактивировать или заменить состав).

### Карточка товара `/dashboard/menu/items/:id`

- Блок **«Состав (ингредиенты)»**:
  - мультивыбор из справочника с поиском (по `name`/`slug/code`) и подсказками;
  - порядок ингредиентов: drag-and-drop (предпочтительно) или стрелки “вверх/вниз”;
  - отдельно для каждой строки состава: флаг `may_contain` и (опционально) `quantity_note` (“по вкусу” / “кусочки” и т.д.).
  - не смешивать с модификаторами и параметрами — отдельная вкладка или секция ниже описания.
- Валидация и UX:
  - не разрешать выбрать один и тот же ингредиент дважды (уникальность `(product_id, ingredient_id)` как источник истины);
  - если ингредиент неактивен, но он уже есть в составе товара: показывать его в списке с пометкой.
- Импорт из текста (фаза 2): вставка строки состава с последующим сопоставлением со справочником и “разбором” в UI для подтверждения.

## Дашборд: филиал

Подробная семантика стопа по позициям и перекрёстные ссылки — в `pages/dashboard/branches/[id]/menu.md`.

Кратко:

- Отдельный блок **«Стоп по ингредиентам»**: список ингредиентов (активных в справочнике), которые **хотя бы у одного товара** встречаются в меню, с переключателем «в стопе». При включении подсвечивается число затронутых позиций (превью).
- При сохранении бэкенд обновляет только `restaurant_ingredient_stops`; витрина и заказ вычисляют итог по правилу выше.

## Заказ и Telegram

- В снимке строки заказа (как для параметров и модификаторов) опционально хранить **текстовый снимок состава** на момент заказа — чтобы смена меню не ломала отображение старых заказов. Минимум: список имён через запятую в JSON или отдельная таблица снимков (когда появится потребность в спорах/аллергиях).
