# SaaS-биллинг PocketMenu (B2B)

Документ описывает модель подписки для владельцев ресторанов и операционные сценарии оплаты модулей платформы (feature toggles).
Контур B2B полностью отделен от клиентских оплат заказов (B2C).

См. общий платежный контекст: `docs/PAYMENTS_RU_YOOKASSA_TBANK.md`.

---

## 1. Цель биллинга

- Монетизировать платформу через подписку (ежемесячная оплата).
- Управлять доступом к функциям на уровне модулей (тумблеров), а не жестких планов.
- Обеспечить предсказуемое поведение при успешной и неуспешной оплате.

---

## 2. Базовые сущности

- `subscription` - агрегированное состояние биллинга `shop` (если нужен общий статус аккаунта).
- `feature_subscription` - подписка `shop` на конкретный модуль (`feature_code`).
- `billing_payment` - платеж по подписке.
- `billing_event` - входящее событие от провайдера (webhook/статус).
- `feature_catalog` - каталог модулей, цен и зависимостей.

Минимальные поля модульной подписки:
- `shop_id`
- `feature_code`
- `status` (`trial`, `active`, `past_due`, `canceled`)
- `current_period_start`, `current_period_end`
- `provider`, `provider_subscription_id` (если есть у провайдера)

---

## 3. Жизненный цикл подписки

### 3.1. Первый платеж

1. Owner выбирает модуль (или пакет модулей как пресет).
2. Платформа создает платеж в B2B merchant-контуре (свои ключи платформы).
3. После webhook `succeeded` модульная подписка становится `active`.
4. Возможности модуля применяются сразу.

### 3.2. Продление (ежемесячно)

Варианты в зависимости от механики провайдера:

- авто-списание (рекуррентный сценарий),
- либо ручная оплата продления по счету/ссылке.

При успешном продлении:
- обновляются `current_period_start/end`,
- сохраняется запись в `billing_payments`.

### 3.3. Неуспешная оплата

Если платеж не прошел:

- подписка переходит в `past_due`;
- включается grace period;
- владелец видит уведомление и CTA на повтор оплаты.

### 3.4. Grace period (продуктовый регламент)

Рекомендуемый вариант для MVP:

- `0-3 день` - все функции доступны, показываются предупреждения;
- `4-14 день` - часть управляющих действий ограничивается (например, интеграции/новые филиалы);
- `15+ день` - read-only режим дашборда или мягкая блокировка по вашему SLA.

Конкретные сроки фиксируются коммерческим регламентом.

### 3.5. Отмена подписки

- по инициативе владельца (на конец оплаченного периода),
- по долгой неуплате (policy-driven cancellation),
- с сохранением истории платежей и событий для аудита.

---

## 4. Upgrade/Downgrade набора модулей

### Upgrade (добавление модулей)

- применяется сразу после успешной доплаты/пересчета;
- новые возможности включаются без ожидания следующего периода.

### Downgrade (отключение модулей)

- рекомендуется применять с начала следующего billing-периода;
- если модуль критичен для текущего процесса, UI должен показывать последствия отключения заранее.

---

## 5. Webhook и источник истины

- Финальное состояние оплаты подписки подтверждается webhook-уведомлением провайдера.
- Redirect/return page используется для UX, но не для фиксации платежного факта.
- Все входящие webhook-события сохраняются с признаком верификации и результатом обработки.

Минимальные требования:
- проверка подписи/секрета;
- идемпотентная обработка дублей;
- журнал ошибок с повторной обработкой (retry).

---

## 6. Отчетность и аудит

Нужно фиксировать:
- кто включил/отключил модуль;
- кто инициировал отмену;
- когда и на какую сумму прошли платежи;
- причины неуспешных списаний (если возвращаются провайдером);
- кто/когда выдал ручной grace override (если допускается).

---

## 7. Связь с коммерческой моделью

Каталог модулей, зависимости и ценовая упаковка описаны в:
- `docs/FEATURE_TOGGLES_PRICING_RU.md`

`SAAS_BILLING_RU.md` определяет платежную механику модулей и их состояний, а не маркетинговые тексты витрины.

---

## 8. Риски и минимальные меры

- Риск: рассинхрон UI и провайдера -> решается webhook-first моделью.
- Риск: повторные события -> идемпотентные ключи и event log.
- Риск: случайная блокировка активного клиента -> staged policy (warning -> limited -> suspended).
- Риск: ручные операционные ошибки -> маскирование секретов, роли Owner-only для критичных действий.

---

## 9. Чеклист готовности B2B-биллинга

- [ ] Подписка хранится на уровне `shop`.
- [ ] Есть отдельный B2B merchant-контур (ключи платформы).
- [ ] Описаны сценарии: first payment, renewal, fail, grace, cancel.
- [ ] Описаны правила upgrade/downgrade.
- [ ] Webhook-модель является источником истины.
- [ ] История платежей и действий доступна для аудита.
