## Что уже есть в коде (факт)

### Backend ЮKassa уже реализован

- Создание платежа: `server/api/checkout/create.post.ts` -> `server/utils/yookassa.ts`.
- Статусы и подтверждение оплаты по webhook: `server/api/webhooks/yookassa.post.ts`.

### UI в настройках организации уже содержит часть нужных полей

На `http://localhost:3001/dashboard/settings/organization` в табе «Контакты и операционные настройки» уже есть:

- Принятие заказов;
- Валюта;
- Часовой пояс;
- Налоговый режим (РФ);
- Реквизиты для публичного футера (Юр. наименование, ИНН, ОГРН/ОГРНИП).

## Чего реально не хватает

### 1) Нет UI для ввода/обновления ключей ЮKassa

Сейчас в админке нет формы для:

- `yookassa_shop_id`;
- `yookassa_secret_key`.

Из-за этого backend готов, но подключение ЮKassa невозможно сделать штатно из интерфейса.

### 2) Нужно разделить вкладку настроек

Текущий таб «Контакты и операционные настройки» перегружен. Нужно вынести в отдельный таб (например, «Оплаты и реквизиты») блоки:

- Принятие заказов;
- Валюта;
- Часовой пояс;
- Налоговый режим (РФ);
- Реквизиты для публичного футера.

### 3) Добавить секцию «Интеграция ЮKassa» в новом табе

Минимальный состав:

- `shopId`;
- Секретный ключ (masked input);
- Статус подключения (подключено/ошибка/не настроено);
- Webhook URL (readonly + copy);
- Кнопка проверки подключения.

Дополнительно (по желанию в MVP):

- Переключатель `test/prod`;
- Технический флаг «онлайн-оплата включена».

### 4) Доработка API сохранения настроек

Существующий поток `organization style/settings` сохраняет `ui_settings` и реквизиты (`legal_name`, `inn`, `ogrn`), но не сохраняет ключи ЮKassa.

Нужно добавить отдельный endpoint (или расширить текущий), который:

- доступен только `Owner`;
- обновляет `shops.yookassa_shop_id` и `shops.yookassa_secret_key`;
- возвращает маскированный секрет в ответе.

### 5) Безопасность

- Не возвращать сырой `secret_key` в UI после сохранения;
- Маскировать ключ в ответах API и интерфейсе;
- Не логировать секреты в открытом виде;
- Для webhook дополнительно добавить валидацию подписи (сейчас подпись читается, но не валидируется).

## Проверка после внедрения

- Владелец может подключить ЮKassa целиком из UI;
- После сохранения повторное открытие страницы показывает корректный статус интеграции;
- Checkout с `payment_method=online` успешно создает платеж;
- Webhook переводит заказ в `paid/canceled/failed` корректно;
- Реквизиты и платежно-операционные поля остаются рабочими после переноса в отдельный таб.
