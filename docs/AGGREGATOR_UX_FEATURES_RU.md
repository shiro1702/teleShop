# Агрегатор: избранное, повтор заказа, фильтры «настроение», отзывы

Документ фиксирует **продуктовую логику** и **черновик технической реализации** для функций на главной города (`/:city_slug`) и связанных сценариев. Опирается на текущий стек: Nuxt, Pinia `cart`, Supabase (`orders`, `orders.items` как JSONB с ценами и модификаторами), `GET /api/client-orders`, `POST /api/order` с пересчётом по `loadTenantProductsForOrder` / `CartItemPayload` (см. `ORDER_SYSTEM.md`, `server/utils/orderLinePricing.ts`).

---

## 1. Избранные рестораны

### 1.1. Цель

- Пользователь **одним тапом** открывает список избранных заведений в текущем городе.
- На карточке ресторана в ленте и на странице ресторана — **иконка «сердце»**: добавить / убрать без лишних шагов.
- Состояние синхронизируется между Web (Supabase-сессия) и TMA (профиль по Telegram), как и заказы.

### 1.2. Данные

| Сущность | Предложение |
|----------|-------------|
| Хранилище | Таблица `restaurant_favorites` |
| Ключи | `customer_profile_id` (FK на `auth.users` / профиль клиента), `restaurant_id` (FK `restaurants`) |
| Дополнительно | `created_at`; опционально `city_id` или денормализованный `city_slug` для быстрых выборок по городу (если один и тот же `restaurant_id` теоретически не дублируется между городами — достаточно фильтра ленты по городу как сейчас) |
| Уникальность | `UNIQUE (customer_profile_id, restaurant_id)` |

**RLS:** строки только владельца профиля; вставка/удаление только свои; чтение списка — свои.

### 1.3. API (черновик)

- `GET /api/client/favorites` — список `restaurant_id[]` или полные карточки для UI (id, name, slug, shop_id, …) с фильтром по `city_slug` из query.
- `POST /api/client/favorites` — тело `{ restaurantId }` — добавить.
- `DELETE /api/client/favorites/:restaurantId` — убрать.
- Или один ресурс `PUT` с телом `{ restaurantIds: string[] }` для редких массовых операций — не обязателен для MVP.

Авторизация: тот же контур, что у `client-orders.get.ts` (`resolveCustomerProfileId`).

### 1.4. UI

- **Главная агрегатора:** в шапке или под поиском — кнопка «Избранное» → переход на `/:city_slug/favorites` (или якорь/модалка со списком). Показывать только если пользователь авторизован; иначе — логин / привязка Telegram (как на checkout).
- **Лента ресторанов:** на карточке — toggle «сердце» с оптимистичным UI и тостом при ошибке (см. правило дашборда для API: для клиентского фронта — тосты, не блокировать страницу).
- **Страница ресторана:** дублировать тот же контрол.

### 1.5. Краевые случаи

- Ресторан скрыт / не в городе — убрать из избранного при открытии списка или показать «недоступен».
- Гость — избранное в `localStorage` с опциональной миграцией после логина (MVP можно только сервер для авторизованных).

## 2. Повтор заказа

### 2.1. Цель

- На главной агрегатора у авторизованного пользователя — блок **последние 3 заказа** (не карусель из 10 — уточнено: три).
- В карточке: **откуда** (название ресторана / бренда), **состав** (строки из позиций, как в истории), **кнопка «Повторить»**.
- По нажатию — **в корзину попадают те же товары** с теми же модификаторами/параметрами, что сохранены в `orders.items`; **цены и доступность пересчитываются на сервере** при оформлении (уже есть в `order.post.ts`). На клиенте при добавлении в корзину — подтянуть актуальные цены из каталога там же, где собирается корзина (единый путь с витриной).

### 2.2. Источник данных

- Расширить ответ `GET /api/client-orders` или завести **`GET /api/client-orders/recent?limit=3&city_slug=...`**:
  - Фильтрация по городу: через join `restaurants` → привязка к городу (как в текущей выдаче агрегатора), чтобы не показывать заказы из другого города.
- Для «повтора» нужны **полные строки заказа**, не только preview: в `orders.items` уже хранится результат после серверного прайсинга (`CartItemPayload` + quantity). Их достаточно, чтобы снова вызвать ту же логику, что при добавлении из меню.

### 2.3. Действие «Повторить»

**Вариант A (предпочтительно):** эндпоинт `POST /api/client/orders/:id/repeat-to-cart`:

1. Проверка: заказ принадлежит профилю.
2. Загрузка `shop_id`, `restaurant_id`, `items`, `fulfillment_type`.
3. Вызов общей функции **валидации и пересчёта строк** (как при создании заказа): удалённые позиции → вернуть `{ ok: false, unavailable: [...] }` или частичный повтор с явным предупреждением в UI.
4. Ответ: `{ ok: true, shopId, restaurantId, fulfillmentType, items: CartItem[] }` в формате, который ждёт `useCartStore` / merge.

**Вариант B:** только клиент: GET отдаёт сырые `items` из заказа; клиент сам маппит в корзину и запрашивает актуальные цены через существующий `products` API — дублирование логики и риск рассинхрона.

Итог: **серверный «repeat»** переиспользует `loadTenantProductsForOrder` и правила стоп-листа/окна меню.

### 2.4. Корзина и контекст

- У корзины уже есть **`scopeKey` / shop** (`stores/cart.ts`). При повторе нужно:
  - переключить контекст на `shop_id` заказа;
  - выставить выбранный ресторан для чекаута (если есть общий composable — использовать его);
  - при непустой корзине другого магазина — модалка «Заменить корзину?» (как при deep link из Telegram bridge).

### 2.5. Отображение на главной

- Блок разместить **выше основной ленты**, после поиска / сториз (согласуется с `pages/[city_slug]/index.md`: «Повторить заказ» как быстрые действия).
- Три карточки: компактно — ресторан, 2–3 строки состава + «ещё N», сумма **как ориентир** с подписью «цены могут измениться».

## 3. Быстрые фильтры «настроение» при поиске ресторана

### 3.1. Цель

- Под строкой поиска на главной — **чипы** (горизонтальный скролл): «Сытное», «Полезное», «Быстро», «Кафе», «Атмосферно», и т.д.
- Набор **зависит от режима** (`Доставка` / `Самовывоз` / `В ресторане`), т.к. сценарий пользователя разный (например, для «в зале» важнее «атмосфера», для доставки — «быстро» и «сытно»).

### 3.2. Модель тегов

**Вариант 1 (гибкий):** справочник `mood_tags` (`id`, `slug`, `label_ru`, `sort_order`, `modes[]` — для каких `listMode` показывать тег).

**Вариант 2 (проще):** enum в коде + колонка `restaurants.mood_tag_slugs text[]` или таблица связей `restaurant_mood_tags (restaurant_id, tag_slug)`.

Заполнение тегов — **в админке ресторана** (дашборд) или вручную в БД на первом этапе.

### 3.3. Логика фильтрации

- Клиент уже получает список ресторанов города (см. `shops.get` / `restaurants.get`). Фильтр:
  - **пересечение** выбранных чипов (AND) или объединение (OR) — зафиксировать в UX: обычно OR («хочу что-то из этого»).
- Поиск по тексту и чипы **совместимы**: сначала текстовая фильтрация, затем по тегам (или наоборот — явно описать в UI).

### 3.4. Пример наборов по режимам

| Режим | Примеры чипов |
|-------|----------------|
| Доставка | Сытное, Полезное, Быстро, Бюджетно, Для компании, Поздний заказ |
| Самовывоз | Быстро, По пути, Экономия на доставке, Кофе с собой |
| В ресторане | Атмосферно, Для свидания, С детьми, Деловой обед, Изысканное |

Часть чипов может повторяться между режимами с разным приоритетом в сортировке.

### 3.5. Будущее

- Персонализация порядка чипов по истории заказов.
- A/B: показ 5 чипов из пула 15.

## 4. Отзывы: ресторан и блюда

### 4.1. Две параллельные системы

| Тип | Для кого | Назначение |
|-----|----------|------------|
| **Отзыв о ресторане** | Публичный рейтинг + модерация | Доверие, негативные сценарии, запрос внешних отзывов при высокой оценке |
| **Оценка блюд (приватная)** | Пользователь + **анонимная аналитика** для кухни | «Не забыть», что зашло; ресторан видит агрегаты по позициям меню без привязки к ФИО |

### 4.2. Триггер: после заказа

- Когда заказ переходит в терминальный успешный статус (например, «выполнен» / «доставлен» — уточнить по `normalizeDashboardStatus`), отправить **push / in-app карточку** «Оцените заказ».
- Одна форма с двумя шагами или двумя блоками: (1) ресторан NPS/звёзды, (2) оценки блюд из `order.items`.

### 4.3. Ресторан: сценарии по оценке

- **1–3 (негатив):** не просить внешние площадки. Показать поле комментария, опционально чекбокс «Связаться со мной», уведомление в дашборд ресторана / саппорт.
- **4–5 (позитив):** после отправки — экран «Спасибо! Оставьте отзыв на Яндекс / 2ГИС / … и получите **баллы**» с пермалинками и таймером акции.
- **Баллы:** интеграция с существующей бонусной системой (`promo`, `loyalty`) — отдельная кампания «reward_external_review» с антифродом (один раз на заказ, проверка по `order_id`).

### 4.4. Блюда

- Таблица `order_item_ratings`: `order_id`, `product_id`, `customer_profile_id`, `rating` 1–5, `created_at`; опционально `note` только для пользователя («острый — ок»).
- Для ресторана — **дашборд агрегатов**: средняя оценка по `product_id`, число оценок, динамика; без списка имён клиентов (анонимность в UI для заведения).

### 4.5. Публичные отзывы о ресторане

- Таблица `restaurant_reviews`: связь с `order_id` (один отзыв на заказ), `rating`, `text`, `status` (`pending|published|rejected`), модерация.
- Отображение на карточке/странице ресторана — только `published`.

### 4.6. Данные и API (черновик)

- `POST /api/client/orders/:id/feedback` — тело: `{ restaurantRating, restaurantComment?, itemRatings: [{ productId, rating }], acceptExternalReviewPrompt? }`.
- Сервер проверяет право (заказ этого пользователя, статус подходящий, не дубль).

## 5. Порядок внедрения (рекомендация)

1. **Повтор заказа** — максимальный эффект на конверсию; опирается на уже существующие `orders` и корзину.
2. **Избранное** — простая таблица + три эндпоинта + UI.
3. **Фильтры настроения** — модель тегов + админка или ручное наполнение + чипы на `index`.
4. **Отзывы** — после стабилизации статусов заказа и политики баллов; сначала MVP без внешних площадок, затем «баллы за отзыв».
