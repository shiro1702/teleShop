# Festival MVP Product Doc

## 1) Цель MVP
Запустить на фестивале рабочий сценарий заказа без очереди через QR, который:
- снижает нагрузку на кассу;
- ускоряет выдачу для гостей;
- дает организаторам измеримую статистику по работе фестиваля;
- формирует кейс для подключения новых точек после события.

## 2) Контекст и ограничения
- Локация: фестиваль еды в одной зоне ТЦ.
- Канал входа: QR-код на веб/миниап.
- Приоритетный режим: только `pickup` (самовывоз).
- Ограничение среды: возможна нестабильная мобильная сеть.

## 3) Границы MVP
## In Scope (в MVP)
- QR-вход в фестивальный режим.
- Фестивальная витрина: список точек и меню.
- Корзина, оплата, создание заказа.
- Статус заказа для гостя: принят -> готов.
- Экран точки/кухни в формате "тапалка" с одной большой кнопкой `Готово`.
- Кнопка `Пауза приема заказов` для точки.
- Отзыв после получения заказа (простая оценка).
- Открытый лидерборд по 2 метрикам:
  - количество пробитых позиций;
  - рейтинг/оценки гостей с порогом минимального числа отзывов.
- Базовая фестивальная аналитика (без ингредиентов).

## Out of Scope (после MVP)
- Интерактивная карта на Leaflet.
- Сложный канбан с drag-and-drop для кухни.
- Продвинутые игровые механики в момент запуска.
- Сложная CRM-автоматизация post-event.

## 4) Роли и основные сценарии
## Гость
1. Сканирует QR на точке или на промо-материалах.
2. Открывает фестивальную витрину, выбирает точку и позиции.
3. Оплачивает заказ.
4. Следит за состоянием заказа.
5. Получает заказ и оставляет оценку.

## Точка (персонал)
1. Видит новый заказ на экране точки.
2. Готовит заказ.
3. Нажимает `Готово`.
4. При перегрузе включает `Пауза приема заказов`.

## Организатор
1. Видит агрегированные показатели фестиваля.
2. Наблюдает открытый лидерборд.
3. Получает финальные цифры для отчета/кейса.

## 5) Приоритизация фич
## P0 (обязательно к запуску)
- QR entry в фестивальный интерфейс.
- Каталог точек + меню.
- Оформление и оплата заказа.
- Статус заказа для гостя.
- Экран точки "тапалка" + `Готово`.
- `Пауза приема заказов`.

## P1 (важно для демонстрации организаторам)
- Открытый лидерборд:
  - "Хит фестиваля" (кол-во позиций),
  - "Народная любовь" (оценки).
- 2 закрытые номинации (подведение в конце):
  - "Турбо-отдача" (скорость),
  - "Железные нервы" (минимум отмен/возвратов).
- Базовый аналитический блок.

## 6) Единые определения метрик и номинаций
## Базовые метрики
- `orders_total`: количество оплаченных заказов.
- `items_total`: сумма всех позиций в оплаченных заказах.
- `gmv_total`: общая сумма оплаченных заказов.
- `avg_prep_time_sec`: среднее время между оплатой и отметкой `Готово`.
- `peak_minute_orders`: максимальное число заказов за 1 минуту.
- `rating_avg`: средняя оценка точки.
- `rating_count`: число оценок точки.
- `unique_buyers_total`: количество уникальных клиентов с хотя бы одним заказом.

## Правила открытого лидерборда
- "Хит фестиваля": сортировка по `items_total`.
- "Народная любовь": сортировка по `rating_avg`, при `rating_count >= 30`.

## Закрытые номинации
- "Турбо-отдача": минимальный `avg_prep_time_sec` при `orders_total >= 30`.
- "Железные нервы": минимальная доля проблемных заказов (отмена/возврат/жалоба).

## 7) KPI MVP для демо организаторам
- Доля заказов через QR среди всех продаж подключенных точек.
- Количество уникальных пользователей, оформивших заказ.
- Общее число заказов и позиций.
- Пиковая нагрузка по минутам.
- Среднее время до статуса `Готово`.
- Средняя оценка и количество отзывов.

## 8) Риски и контрмеры
## Риск: Нестабильная сеть
- Контрмера: облегченный фестивальный интерфейс, минимальный вес страницы, fallback-сообщения о переподключении.

## Риск: Точка перегружена
- Контрмера: `Пауза приема заказов`, ручное управление доступностью точки.

## Риск: Спорные оплаты/пропавший заказ
- Контрмера: onsite-поддержка и регламент ручной проверки/подтверждения.

## Риск: Низкая доля QR-заказов
- Контрмера: отдельная зона "выдача app-заказов", явный CTA на стойках, мотивация через лидерборд.

## 9) Критерий успеха MVP
MVP считается успешным, если на фестивале стабильно работает полный цикл:
QR -> заказ -> оплата -> выдача -> отзыв, а организатор получает измеримые результаты по скорости и объему заказов через систему.
