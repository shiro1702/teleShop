# Архитектура и интеграция Фестиваля с Агрегатором

В этом документе описывается техническая реализация фестивального режима в контексте существующей платформы-агрегатора (Nuxt, Pinia, Supabase), чтобы изолировать данные, управлять нагрузкой и обеспечить бесшовный переход пользователей в "City Mode" после завершения мероприятия.

## 1. Структура БД: Изоляция фестивальных данных
**Проблема:** Нельзя просто смешать фестивальные рестораны с городской выдачей, так как это испортит опыт обычным пользователям. Кроме того, фестивальные точки могут быть "поп-ап" версиями существующих заведений с другим (сокращенным) меню.

**Решение (Гибридное):**
1. **Сущность `Festival` (Фестивальная зона):**
   - Можно создать отдельный фейковый "Город" (например, `city_slug = "festival-2026"`), но идеологически это событие ВНУТРИ города.
   - Лучший подход: Добавить таблицу `festivals` (id, city_id, name, slug, start_date, end_date) и связующую таблицу `festival_restaurants` (festival_id, restaurant_id).
2. **Изоляция меню (Tenant Management):**
   - Если ресторан участвует в фестивале с обычным меню, мы используем его текущий `restaurant_id`.
   - Если меню урезанное (только 3 позиции), создаем **скрытый клон** ресторана (новый `shop_id` или `restaurant_id` со специальным тегом `is_popup: true`), который привязан только к фестивалю.
3. **Хранение достижений (Ачивки):**
   - `festival_achievements` (id, profile_id, festival_id, achievement_type, unlocked_at).
   - После фестиваля данные отсюда конвертируются в `customer_profiles.status` (City Mode).

## 2. Интеграция с клиентским приложением (Aggregator UX)
Как гость попадает в "Фестивальный режим"?

1. **Вход через QR-код (Основной сценарий):**
   - QR-коды на столах/афишах ведут на `https://app.com/festival/fest-slug`.
   - При переходе по этой ссылке приложение (через Pinia/Cookies) запоминает `current_festival_id`.
   - Интерфейс меняется: пропадает выбор доставки, скрываются городские фильтры, появляется блок "Сториз" и "Лидерборд", а в ленте остаются ТОЛЬКО корнеры, привязанные к этому `festival_id`.
2. **Вход через главную страницу города (`/:city_slug`):**
   - На главной странице города (для тех, кто зашел сам, а не по QR) во время проведения фестиваля висит большой **Баннер-шторка**: "Мы на фестивале X! Заказывай без очередей ➔ Переключиться в режим фестиваля".
3. **Сценарий Onboarding (Авторизация в толпе):**
   - Гость хочет есть сейчас, у него нет времени на сложную регистрацию.
   - Обязательно: Вход в 1 клик через уже внедренную авторизацию Telegram внутри Mini App.
   - Для Mini App в мессенджере MAX: тот же принцип быстрого входа (единый профиль клиента), без отдельного тяжелого шага регистрации на фестивале.

## 3. Нагрузочное тестирование и кэширование (Highload)
**Проблема:** Фестиваль — это резкие спайки трафика (сотни сканирований QR-кодов в минуту), когда ведущий со сцены говорит "Заказывайте еду в приложении!". Self-hosted Supabase (Reg.ru) может не выдержать пиковое количество чтений без промежуточного кэша.

**Решение (Reg.ru + self-hosted Supabase):**
1. **Кэширование на уровне backend/API:**
   - Список корнеров и их меню (`GET /api/festival/restaurants`) отдавать через server-side кэш (Redis при наличии или in-memory TTL cache в Nuxt/Nitro как fallback).
   - TTL для меню: 30-120 секунд (с ручным invalidation при обновлении ассортимента).
2. **Кэш для фестивальных виджетов:**
   - Данные "Пульса фестиваля" кэшировать на 30-60 секунд, чтобы не дергать Supabase на каждый просмотр.
   - Обновление агрегатов делать батчами (cron/job), а не прямыми запросами на каждый клиентский рендер.
3. **Асинхронные сокеты (B2B):**
   - Для экранов поваров использовать Supabase Realtime только для подписки на `orders` с фильтром `shop_id=eq.X`.
   - Для клиентских экранов с высокой посещаемостью предпочесть polling/кэшируемые API-эндпоинты вместо массовых realtime-подписок.

## 4. Оффлайн-проблемы и связь на площадке
- При большом скоплении людей мобильный интернет ложится.
- **Для B2B (Рестораны):** Организаторы должны обеспечить закрытую Wi-Fi сеть только для планшетов корнеров. Иначе они не увидят заказы.
- **Для B2C (Гости):** PWA (Service Workers) для кэширования статики приложения, чтобы загружались только JSON-данные. Уведомление "Заказ готов" дублировать через Telegram-бота (SMS слишком дорогие и могут задерживаться).

## 5. Выход из Фестиваля (Этап 3 - Post-Processing)
Когда `end_date` фестиваля проходит:
1. Роут `/festival/fest-slug` закрывается или показывает страницу "Фестиваль окончен! Вот твои итоги".
2. Флаг `current_festival_id` в стейте клиента очищается, возвращая его на стандартный `/:city_slug` с полным списком заведений города.
3. Запускаются cron-джобы на отправку персонализированных пушей (удержание) и начисление баллов.
4. "Скрытые клоны" фестивальных ресторанов деактивируются (`is_active: false`), чтобы они случайно не всплыли в городском поиске на следующий день.
