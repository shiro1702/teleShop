# План интеграции TeleShop с Quick Resto API

Документ описывает, что реалистично и безопасно интегрировать в TeleShop на базе текущего функционала: мультитенантность (`shop`/`restaurant`), витрина, checkout, заказы, маркетинг и feature toggles.

Связанные документы:
- `docs/FEATURE_TOGGLES_PRICING_RU.md`
- `docs/MULTI_TENANT_SAAS.md`
- [Quick Resto API](https://quickresto.ru/api/)

---

## 1) Короткий ответ по возможностям

### 1.1. Запрашивать и обновлять меню
Да, это приоритет №1 и самый безопасный первый контур.

Что делаем:
- импорт номенклатуры из Quick Resto в ваше меню (`products`, категории, цены, стоп-листы/доступность);
- двусторонняя синхронизация только после стабилизации (сначала one-way: Quick Resto -> TeleShop);
- по филиалам: использовать связку `shop_id + restaurant_id`, чтобы не смешивать меню между ресторанами;
- запуск синхронизации меню в MVP вручную из дашборда (кнопка), без cron-обновлений.

### 1.2. Создавать заказы
Да, это приоритет №2, но через staged rollout.

Что делаем:
- создание заказа в TeleShop как сейчас + отправка в Quick Resto;
- хранение внешнего идентификатора (`external_order_id`) и статуса обмена;
- ретраи, idempotency-key, журнал событий интеграции;
- маппинг статусов Quick Resto -> внутренние статусы заказа.

### 1.3. Синхронизировать бонусную систему ресторана
Да, возможно, но как второй этап после стабильного меню+заказов.

Что делаем:
- сначала read-only: получение баланса/правил списания;
- затем write-flow: резерв/списание при оформлении заказа, откат при отмене;
- синхронизация только в рамках конкретного ресторана/тенанта (никакого общего кошелька между разными `shop`).

### 1.4. Синхронизировать промокоды Quick Resto
Да, и это нужно закладывать отдельным контуром рядом с лояльностью.

Что делаем:
- read-flow: подтягиваем правила промокодов (активность, сроки, ограничения);
- checkout-flow: серверная валидация промокода в Quick Resto перед подтверждением заказа;
- write-flow: фиксируем применение промокода в заказе и синхронизируем итоговую скидку;
- определяем приоритет скидок: промокод vs бонусы (правило «что применяем первым» едино на backend).

## 2) Принципы интеграции для вашей архитектуры

1. **Tenant isolation first**  
   Любой обмен с Quick Resto выполняется в контексте `shop_id`, а операции филиала — в контексте `restaurant_id`.

2. **Source of truth по этапам**  
   - Этап 1: Quick Resto — источник правды по меню.  
   - Этап 2: TeleShop — источник правды по UX заказа; Quick Resto — источник учёта/операционного статуса.

3. **Не блокировать UX при сбое интеграции**  
   Ошибки внешнего API не должны ломать страницу: показываем мягкие ошибки, сохраняем retry-задачу, логируем.

4. **Feature toggle-подход**  
   Интеграцию вынести в отдельные модули биллинга (см. `FEATURE_TOGGLES_PRICING_RU.md`).

## 3) Предлагаемые платные модули (тумблеры)

1. `integration_quickresto_menu_sync`
   - импорт меню, цен, доступности;
   - режимы: ручной sync (MVP), расписание опционально позже.

2. `integration_quickresto_order_push`
   - отправка заказов в Quick Resto;
   - статусы обмена, ретраи, мониторинг.

3. `integration_quickresto_loyalty_bridge`
   - баланс бонусов, применение/отмена бонусов по заказу;
   - история операций по клиенту.

4. `integration_quickresto_promocode_sync`
   - синхронизация и валидация промокодов Quick Resto;
   - применение скидок в checkout с серверной проверкой.

Зависимости:
- `integration_quickresto_order_push` требует `crm_orders_db`;
- `integration_quickresto_loyalty_bridge` требует `crm_orders_db` + `marketing_loyalty`.
- `integration_quickresto_promocode_sync` требует `crm_orders_db` + `marketing_loyalty`.

## 4) Поэтапный план внедрения

## Этап 0. Discovery + контракт API (2-4 дня)
- Проверить доступные endpoint’ы Quick Resto по факту доступа (меню, стоп-листы, заказы, лояльность, webhooks).
- Зафиксировать auth-модель, rate limits, SLA, формат ошибок.
- Подготовить таблицу маппинга полей `Quick Resto <-> TeleShop`.

Результат:
- `docs/integrations/quickresto-api-contract.md` (таблица endpoint’ов и полей);
- список ограничений для MVP.

## Этап 1. База интеграции и хранение секретов (1 неделя)
- Добавить сущность подключения Quick Resto на уровне `shop` (и override на `restaurant` при необходимости).
- Реализовать серверный health-check подключения.
- Добавить аудит: кто подключил, когда, какой филиал привязан.

Что уже есть в схеме и переиспользуем:
- `shops.integration_keys` — хранение ключей интеграции на уровне бренда (`shop`);
- `products.external_id`, `categories.external_id`, `modifier_groups.external_id`, `modifier_options.external_id` — стабильный маппинг сущностей меню;
- `restaurant_product_overrides.is_in_stop_list` — стоп-листы на уровне филиала;
- `shop_promo_codes`, `promo_code_uses`, `shop_customer_balances`, `loyalty_ledger` — готовая база для промо/бонусов;
- `payment_webhook_events` — паттерн идемпотентной обработки внешних webhook-событий.

Что добавить минимально:
- `quickresto_sync_jobs` (job type, status, started_at, finished_at, attempts, initiated_by);
- `quickresto_events` (event_type, external_event_id unique, payload, processed_at, error).

## Этап 2. Синхронизация меню (MVP) (1-2 недели)
- One-way импорт из Quick Resto в TeleShop по кнопке «Синхронизировать меню».
- Маппинг: категории, позиции, варианты, модификаторы, цены, доступность.
- Безопасный upsert по `external_id`.
- Режим dry-run (показать diff без применения).
- Защита от параллельных sync: только 1 активный job на `shop_id` (advisory lock / unique partial index).
- После полного sync пересобираем `restaurant_product_overrides` только для связанных `product_id`, чтобы не потерять ручные скрытия/оверрайды.

### Этап 2.1. Стоп-листы через webhook (если поддерживается Quick Resto) (2-4 дня)
- Поднять endpoint webhook для событий availability/stop-list.
- Обновлять `restaurant_product_overrides.is_in_stop_list` точечно по `restaurant_id + product_id`.
- Если webhook недоступен со стороны Quick Resto: fallback — кнопка «Обновить стоп-листы» в том же разделе.

Acceptance:
- ресторан может запустить sync и увидеть актуальное меню в витрине;
- не теряются локальные поля TeleShop (контент/оформление), если они не приходят из Quick Resto;
- стоп-листы после webhook меняются на сайте почти сразу (или через ручной fallback без полного sync меню).

## Этап 3. Отправка заказов в Quick Resto (1-2 недели)
- При создании заказа в TeleShop отправлять заказ во внешнюю систему.
- Хранить `external_order_id`, `external_status`, `last_sync_error`.
- Реализовать outbox/queue + retry policy.
- Добавить ручное действие «переотправить заказ».

Acceptance:
- заказ из checkout стабильно попадает в Quick Resto;
- при сбое есть прозрачно видимый статус и повторная отправка без дублей.

## Этап 4. Лояльность/бонусы (этап Growth) (1-2 недели)
- Read-only интеграция баланса бонусов по клиенту.
- Применение бонусов при checkout с серверной валидацией.
- Компенсационные операции при отменах/рефандах.

Acceptance:
- списание бонусов синхронно отражается в обеих системах;
- невозможен двойной расход бонусов при повторных запросах.

## Этап 5. Промокоды Quick Resto (этап Growth) (1 неделя)
- Импорт/кеширование активных промокодов и их ограничений.
- Валидация промокода при checkout через сервер (не доверять только фронту).
- Маппинг скидки в структуру заказа TeleShop + внешний заказ Quick Resto.
- Обработка кейсов «промокод истек/недоступен для филиала/не совместим с бонусами».

Acceptance:
- пользователь видит прозрачную ошибку, если промокод невалиден;
- применённый промокод одинаково отражается в TeleShop и Quick Resto;
- правила совместимости с бонусами выполняются детерминированно.

## 5) Что делаем в UI/UX (дашборд ресторана)

Новый раздел: `/dashboard/integrations/quickresto`

Блоки:
1. Статус подключения (подключен/ошибка/не настроен).
2. Привязка филиалов (`restaurant`) к внешним точкам Quick Resto.
3. Кнопки:
   - «Проверить подключение»
   - «Синхронизировать меню»
   - «Переотправить проблемные заказы»
   - «Синхронизировать промокоды»
4. Журнал последних событий интеграции.

Важно:
- В пользовательских ошибках давать понятные сообщения без технического шума.
- Любая длительная операция — асинхронная задача со статусом.

## 6) Риски и как их закрыть

1. **Дубли заказов при сетевых сбоях**  
   Решение: idempotency key + outbox + уникальный индекс на `external_order_id` + таблица dedup ключей по `shop_id`.

2. **Конфликт меню между системами**  
   Решение: на MVP фиксируем source of truth (Quick Resto), локальные правки только в разрешённых полях; upsert только по `external_id`.

3. **Потеря тенант-изоляции**  
   Решение: обязательные проверки `shop_id/restaurant_id/city_id` в каждом серверном endpoint интеграции + RLS-safe запросы для dashboard контуров.

4. **Расхождение бонусного баланса**  
   Решение: использовать существующий `loyalty_ledger` как источник правды + компенсационные транзакции + сверка по расписанию.

5. **Неверное применение промокодов и конфликт с бонусами**  
   Решение: единый server-side discount engine, приоритеты скидок в конфиге ресторана, запись `promo_snapshot` в заказ и обязательная валидация через Quick Resto.

6. **Параллельные ручные синхронизации ломают данные**  
   Решение: lock на запуск sync по `shop_id`, статус активной операции в UI и запрет повторного старта до завершения.

7. **Поддельные или повторные webhook события по стоп-листам**  
   Решение: верификация подписи webhook + dedup по `external_event_id` + хранение сырых payload в `quickresto_events`.

## 7) Технический минимум для старта реализации

Backend:
- `POST /api/dashboard/integrations/quickresto/connect`
- `POST /api/dashboard/integrations/quickresto/health-check`
- `POST /api/dashboard/integrations/quickresto/menu-sync`
- `POST /api/dashboard/integrations/quickresto/stoplist-sync` (manual fallback)
- `POST /api/dashboard/integrations/quickresto/orders/retry-failed`
- `POST /api/dashboard/integrations/quickresto/promocodes-sync`
- `POST /api/checkout/quickresto/validate-promocode`
- `POST /api/webhooks/quickresto/menu-availability` (если Quick Resto поддерживает webhook)

Workers/cron:
- `quickresto:order-dispatch`
- `quickresto:loyalty-reconcile` (после запуска бонусов)
- `quickresto:promocode-sync` (опционально, если понадобится фоновая сверка)

Data model:
- во всех таблицах интеграции обязательны `shop_id` и при необходимости `restaurant_id`.
- использовать уже существующие `external_id` в меню и `promo_snapshot` в `orders`;
- для webhook-событий хранить `external_event_id` с уникальным индексом.

## 8) Приоритизация по ценности для бизнеса

P0 (сразу):
- Подключение интеграции + health-check.
- Импорт меню (one-way, manual sync по кнопке).
- Быстрое обновление стоп-листов (webhook или manual fallback).

P1:
- Отправка заказов + мониторинг ошибок.

P2:
- Бонусная система (баланс, списание, откат).
- Промокоды Quick Resto (валидация и применение).

Итог для продаж:
- Быстрый value: «меню и стоп-листы синхронизируются автоматически».
- Операционный value: «заказы уходят в учетную систему без ручного дубляжа».
- Retention value: «бонусы синхронны между витриной и кассовым контуром ресторана».
- Promo value: «акции и промокоды ресторана работают в витрине без ручного дублирования».

## 9) Definition of Done для интеграции Quick Resto (MVP+)

- Для выбранного ресторана меню обновляется из Quick Resto без ручной миграции.
- Заказ из TeleShop создаётся в Quick Resto и имеет связанный внешний id.
- Ошибки интеграции не ломают оформление заказа и видны в дашборде.
- Все операции проходят в tenant-контексте (`shop`/`restaurant`) без утечек данных.
- Бонусы (если включен модуль) корректно резервируются/списываются/возвращаются.
- Промокоды Quick Resto (если включен модуль) валидируются и применяются консистентно в обеих системах.

## 10) Что еще важно учесть под текущую архитектуру TeleShop

1. **Связка филиалов с внешними точками**  
   Нужна явная таблица маппинга `restaurant_id <-> quickresto_place_id`, иначе стоп-листы и остатки некуда точно применять.

2. **Версионирование снимка меню**  
   Для отладки полезно хранить `last_menu_sync_hash` и `last_menu_synced_at` на интеграции: видно, что реально изменилось между sync.

3. **Инвалидация кэша витрины**  
   После menu-sync и stop-list webhook нужен явный purge/invalidate ключей витрины, иначе пользователь увидит старые данные.

4. **Стратегия недоступности Quick Resto в checkout**  
   Нужен продуктовый флаг на уровне ресторана: `strict_mode` (не даем оформить заказ без внешней валидации) или `grace_mode` (разрешаем заказ с пометкой manual confirm).

5. **Наблюдаемость и support-debug**  
   В дашборде нужны correlation id, время последнего успешного sync и удобный экспорт ошибок, чтобы поддержка быстро разбирала инциденты.
