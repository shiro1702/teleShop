# План: кеш `/api/restaurants` и сохранение режима получения заказа

Документ фиксирует выявленные разрывы в текущей реализации и предлагаемые правки. Без изменения продуктовых требований: уважать явный выбор пользователя (самовывоз / доставка / в зале) при переходах между страницами города, витрины ресторана и чекаута.

---

## Часть A. Дублирование запроса `GET /api/restaurants`

### Контекст

Один и тот же запрос (`/api/restaurants?shop_id=…`) выполняется на витрине (`pages/index.vue`, `loadRestaurantModes`) и в чекауте через `useCheckoutTenantRestaurants` (`loadRestaurants`). Внутри composable есть дедупликация по ключу магазина и in-flight promise, но **каждый экземпляр страницы / другой входной точки** делает свой `$fetch`: при навигации витрина ↔ чекаут часто два сетевых запроса.

Эндпоинт: `server/api/restaurants.get.ts` — данные завязаны на `shop_id` и настройки организации; ответ можно кешировать с коротким TTL с учётом заголовков тенанта.

### Цели

- Уменьшить число повторных запросов на один и тот же магазин в одной вкладке.
- Не отдавать устаревшие данные на долгий срок (настройки филиалов и org могут меняться).

### Предлагаемые шаги

1. **Клиент (Nuxt)**  
   - Вынести загрузку списка филиалов в общий слой с ключом **`shop_id` (+ при необходимости признак Telegram initData, если он влияет на ответ)**.  
   - Варианты: `useAsyncData` / `useFetch` с одинаковым `key` (например `restaurants:${shopId}:${initFlag}`), либо `useState` для результата и общей функции `loadRestaurants`.  
   - Убедиться, что заголовки (`x-shop-id`, auth для мессенджера) совпадают между вызовами — иначе кеш разойдётся.

2. **HTTP-кеш (опционально)**  
   - Для публичного GET выставить разумный `Cache-Control` (например десятки секунд или 1–2 минуты) или использовать Nitro `defineCachedEventHandler` с ключом, включающим **`shop_id`** и параметры, влияющие на ответ.

3. **Инвалидация / TTL**  
   - Короткий TTL из-за полей «на сейчас» (`effective_working_hours` и т.д.).  
   - После изменений в админке допустима задержка до истечения TTL — при необходимости позже добавить тег инвалидации или версию настроек в ключ.

### Файлы для правок

- `composables/useCheckoutTenantRestaurants.ts` — общий источник данных или вызов обёрнутого composable.  
- `pages/index.vue` — `loadRestaurantModes` перевести на общий источник.  
- При серверном кеше: `server/api/restaurants.get.ts` или `nuxt.config` (`routeRules`).

## Часть B. Режим получения заказа (fulfillment): проблемы

### B1. Город и витрина не связаны

На `pages/[city_slug]/index.vue` выбранный режим списка (`listMode`: `delivery` | `pickup` | `dine-in`) сохраняется в:

`localStorage`: `teleshop-city-fulfillment:${city_slug}`

Этот ключ **нигде не читается** при входе на витрину ресторана. На `pages/index.vue` старт — **`delivery`**, затем подмешивается только `teleshop_checkout_state:${scope}` для **конкретного магазина**. Итог: пользователь выбрал «самовывоз» на городе → зашёл в ресторан впервые → может увидеть **доставку** по умолчанию.

### B2. Приоритет доставки при несовпадении на витрине

В `loadRestaurantModes` (finally), если текущий тип не входит в `derivedAllowedFulfillmentTypes`, выставляется:

`allowed.includes('delivery') ? 'delivery' : allowed[0]`

Это усиливает ощущение «подставили доставку», даже когда пользователь явно выбирал другой канал на предыдущем шаге.

### B3. Чекаут: восстановление типа до загрузки филиалов

В `pages/checkout.vue`, `restoreFromPlainObject` применяет `fulfillmentType` только если:

`availableFulfillmentTypes.value.includes(obj.fulfillmentType)`

Пока `restaurantsLoaded === false`, в `useCheckoutTenantRestaurants` **`availableFulfillmentTypes` пустой** → сохранённый самовывоз **не применяется**. Состояние остаётся дефолтным `fulfillmentType: 'delivery'`.

После загрузки филиалов срабатывает watch в composable: если текущий тип не в списке, подставляется **`types[0]`** (порядок из `Set`, не предпочтение пользователя).

### B4. `qr-menu` при восстановлении

В `restoreFromPlainObject` условие восстановления типа включает только `delivery` и `pickup`; **`qr-menu` из сохранённого JSON не восстанавливается** этим блоком.

## Часть C. Режим получения заказа: предлагаемая логика

### C1. Явное «предпочтение пользователя»

- Хранить отдельно от технических сбросов (например ключ вида `teleshop-user-fulfillment-intent:${shopId}` или расширить существующий checkout-state осмысленным полем `preferredFulfillmentType`).
- Обновлять **при явном выборе** в селекторе на витрине и в чекауте, не перезатирать при каждом внутреннем пересчёте.

### C2. Связка город → витрина

- При монтировании витрины (есть `city_slug` и `tenant_slug` в маршруте): прочитать `teleshop-city-fulfillment:${city_slug}`.
- Смаппить: `delivery` → `delivery`, `pickup` → `pickup`, `dine-in` → `qr-menu` (или тот тип, который соответствует продукту для «в зале»), **только если этот тип доступен** у ресторана после загрузки `/api/restaurants`.
- Если городской режим недоступен для точки — fallback по правилам ниже.

### C3. Единый порядок fallback (витрина и чекаут)

1. Если сохранённое предпочтение (checkout-state или intent) **входит** в `allowed` — использовать его.
2. Иначе если доступен режим с **городской страницы** и он в `allowed` — использовать его.
3. Иначе — текущая политика по умолчанию (например первый из фиксированного приоритета `pickup` → `delivery` → … или как решите продуктово), **не автоматически доставка**, если пользователь явно выбирал самовывоз и он есть в `allowed`.

### C4. Чекаут: порядок инициализации

- После **`loadRestaurants()`** повторно вызвать применение сохранённого состояния для `fulfillmentType` **или** хранить «сырой» сохранённый тип и применять его при первом **ненулевом** `availableFulfillmentTypes`.
- Расширить `restoreFromPlainObject`: восстанавливать и **`qr-menu`**, если он в `allowed`.
- Подправить watch на `availableFulfillmentTypes`: при несовместимости текущего типа подставлять не `types[0]`, а **сохранённое предпочтение из localStorage**, если оно допустимо; иначе — согласованный fallback.

### C5. Согласованность ключей storage

- Уже используются `buildCheckoutStorageKey` / `resolveCartScopeKey` — убедиться, что при смене маршрута (город + тенант) предпочтение не теряется из-за другого scope; при необходимости продублировать intent в ключе, привязанном к `shop_id`.

## Часть D. Чеклист файлов

| Область | Файлы |
|--------|--------|
| Город | `pages/[city_slug]/index.vue` (уже пишет `teleshop-city-fulfillment`) |
| Витрина | `pages/index.vue` — инициализация `selectedFulfillmentType`, `loadRestaurantModes` |
| Чекаут | `pages/checkout.vue` — `restoreFromPlainObject`, `onMounted` порядок, при необходимости watch после загрузки |
| Composable | `composables/useCheckoutTenantRestaurants.ts` — `availableFulfillmentTypes` watch, опционально общая загрузка ресторанов |
| API (кеш) | `server/api/restaurants.get.ts` и/или `nuxt.config` |

## Часть E. Проверка (UAT)

1. Город: выбрать самовывоз → открыть ресторан с доставкой и самовывозом → витрина должна открыться в **самовывозе** (если доступен).  
2. Витрина: выбрать самовывоз → перейти в корзину/чекаут → режим **самовывоз**, без подмены на доставку после загрузки филиалов.  
3. Обновление страницы чекаута с сохранённым `pickup` в localStorage → после загрузки API режим остаётся **pickup**.  
4. Режим «в зале» (`qr-menu`): сохранение и восстановление после перезагрузки.  
5. Регрессия: ресторан без самовывоза → корректный fallback и отсутствие сломанного UI.

---

## Примечание по объёму работ

Части A и B–C можно внедрять **по отдельности**: кеш ресторанов не блокирует правку fulfillment; правка fulfillment может потребовать небольшого рефакторинга composable и порядка `onMounted` на чекауте.
