# План: лишние запросы `/api/tenant` при навигации внутри витрины ресторана

## Контекст

При переходах между страницами одного тенанта (`/{city}/{tenant}/cart`, `/checkout`, `/bonuses`, `/orders` и т.д.) не нужно каждый раз заново загружать профиль ресторана (тема, юрданные, часы и т.п.). Достаточно загрузить при первом входе в контекст этого тенанта и при смене ресторана / филиала.

Источник логики: [`composables/useTenant.ts`](../../composables/useTenant.ts) — `useState('tenant-state')`, `loadTenantSettings()`, watchers.

## Цель

- Убрать повторные клиентские запросы к `/api/tenant?shop_id=…` при SPA-переходах внутри одного `tenant_slug` при неизменных значимых query-параметрах.
- Сохранить корректную перезагрузку при смене ресторана (`tenant_slug` / `shop_id`) и при смене контекста филиала (`branch_id`, `restaurant_id`), если они используются в продукте.

## Этапы работы

### (выполено!) Этап 1 — watcher по `branch_id` / `restaurant_id`

**Проблема:** геттер `() => [route.query.branch_id, route.query.restaurant_id]` возвращает новый массив при каждом запуске; при смене маршрута возможны ложные срабатывания и сброс `loaded`, из-за чего снова вызывается `/api/tenant`.

**Сделать:**

- Переписать наблюдение так, чтобы колбэк выполнялся только при **реальном изменении** значений `branch_id` и `restaurant_id` (два scalar-watch, один watch с явным сравнением предыдущего и текущего — по соглашению проекта).
- Если филиал/ресторан в query поменялись — перезагрузить tenant-контекст, если бэкенд от этого зависит; если не менялись — не сбрасывать `loaded` и не вызывать API.

---

### Этап 2 — явный «scope» тенанта в состоянии

**Идея:** помимо флагов `loaded`/`loading` хранить ключ последней успешной загрузки, например `resolvedTenantKey` (значение `tenantKey` на момент `applyTenant`).

**Сделать:**

- Вызывать `loadTenantSettings()`, если контекст ещё не загружен **или** текущий `tenantKey` отличается от сохранённого после последней успешной загрузки **или** позже понадобится явный «force refresh».
- При смене `tenantKey` по-прежнему сбрасывать поля предыдущего ресторана и подгружать заново (согласовать с существующим watcher на `tenantKey`).

---

### Этап 3 (опционально) — HTTP-кэш для `/api/tenant`

**Сделать:** в [`server/api/tenant.get.ts`](../../server/api/tenant.get.ts) настроить заголовки кэширования (`Cache-Control`, при необходимости ETag), согласовав с тем, есть ли в ответе данные, завязанные на пользователя (`private` vs `public`), чтобы случайный лишний вызов давал меньше нагрузки.

---

### Этап 4 — данные тенанта в ответе страницы (payload / Nitro middleware)

Уже частично есть через `event.context.tenant` на сервере; можно дополнять контекст **полным набором полей** (как в `/api/tenant`), чтобы не сбрасывать `loaded` и не дергать API после гидрации.

Альтернатива или дополнение: передавать снапшот через `useAsyncData` на layout уровня `[city_slug]/[tenant_slug]` и синхронизировать с `useTenant` после гидрации.

**Сделать (на выбор архитектуры):**

- Расширить серверный tenant-контекст в middleware/handlers так, чтобы он совпадал по полезной нагрузке с `/api/tenant`, и в `useTenant` при SSR применять его без принудительного клиентского повторного запроса (убрать или сузить место, где сейчас после `applyTenant` выставляется `loaded = false` только из‑за неполного контекста).
- Либо завести layout тенанта и загружать тот же снапшот через `useAsyncData` с ключом по `city_slug` + `tenant_slug`, затем прокидывать в общее состояние.
