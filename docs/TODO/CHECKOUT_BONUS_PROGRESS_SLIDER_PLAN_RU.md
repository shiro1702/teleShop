# Прогресс-бар и ползунок списания бонусов на checkout

Документ для постановки задачи: **визуально согласованный** с остальным оформлением заказа контрол выбора суммы списания бонусов — «трек» с заполнением (как у доставки) и **ползунок от минимума к максимуму**, при этом **бизнес-логика списания уже реализована** на сервере и в превью; основной объём работы — UI/UX и согласованность с `promo/preview`.

## Цель

- Пользователь на этапе оформления заказа **одним жестом** выбирает, сколько бонусов списать: от **0** (не списывать) до **максимума, разрешённого правилами** (баланс, % от субтотала после промо, субтотал).
- Визуально: **прогресс-бар** (заполненная часть = доля выбранного списания от максимума) + **движок (thumb)**, перетаскиваемый вдоль трека. Подписи: текущее значение, максимум, по желанию — эквивалент в ₽ (1 бонус = 1 ₽ к оплате товаров — как сейчас в расчётах).
- Для `checkout` источник истины по максимуму списания — серверный расчёт (`preview`/`order`), а UI только отображает доступный диапазон и текущее значение.

## Объём работ (фронтенд)

1. **Заменить или обернуть** нативный `range` в компонент с:
   - визуальным **треком** и **прогресс-заливкой** под бренд (primary / tenant);
   - **thumb** поверх трека;
   - при необходимости — **touch-friendly** высота hit-area (≈44px по гайдлайнам).

2. **Диапазон ползунка**
   - `min = 0`, `max = maxBonusAvailable` (как сейчас).
   - При изменении корзины, промокода или ответа превью `watch(maxBonusAvailable)` уже подрезает `bonusToSpend` и дергает превью — сохранить это поведение.

3. **События и превью**
   - Сейчас превью вызывается на `@change` у `range` (отпускание). Для плавного перетаскивания рассмотреть:
     - **`input`** (каждый шаг) — обновлять локальное отображение суммы сразу;
     - **`change`** или **debounced** вызов `runPromoPreview` — чтобы не бомбить `/api/promo/preview` на каждый пиксель (см. также [`docs/TODO/CART_CHECKOUT_FETCH_OPTIMIZATION_PLAN_RU.md`](./CART_CHECKOUT_FETCH_OPTIMIZATION_PLAN_RU.md)).
   - Инвариант: итоговые `payableGoods`, строка «Списано бонусов», итог заказа совпадают с ответом превью после последнего запроса.

4. **Доступность**
   - Сохранить семантику для скринридеров: связь подписи «Списать бонусов» с контролом, `aria-valuemin`, `aria-valuemax`, `aria-valuenow`, `aria-valuetext` (например «150 бонусов из 400»).
   - Клавиатура: шаг ползунка `step="1"`; при кастомном UI продублировать поведение или оставить нативный input скрытым / связанным.

5. **Состояния**
   - Не авторизован / баланс не загружен: блок бонусов скрыт или с сообщением (как сейчас по `isAuthorizedForOrder` и `loyaltyBalance`).
   - `bonusesEnabled === false`: текст про отключение (уже есть).
   - `maxBonusAvailable === 0`: ползунок не показывать или disabled с пояснением.

6. **Единый визуальный язык**
   - Согласовать радиусы, цвета, типографику с карточкой промо и блоком `deliveryProgress` на том же шаге.

7. **Границы ответственности (важно для мультитенанта)**
   - Не дублировать во фронте сложные правила категорий/исключений; клиент получает уже рассчитанный сервером `maxBonusForOrder`.
   - Если для ресторана активен внешний провайдер лояльности (например, iiko), UI использует только лимиты/значения из текущего preview-ответа.
   - Ползунок не должен пытаться учитывать "будущие" правила (уровни, срок сгорания, welcome-ограничения) локальной логикой.

## Объём работ (бэкенд) — по необходимости

- **MVP по API не требует изменений**: `preview` и `order` уже принимают `bonusPointsToSpend`.
- Дополнительно (не обязательно для первой итерации): вынести переиспользуемый **маленький превью только лояльности** или объединить с другими запросами — только если профилирование покажет узкое место (см. оптимизационный документ выше).

## Критерии приёмки

1. Ползунок двигается от **0** до **фактического максимума** для текущей корзины и промо; при смене промокода или состава корзины максимум и позиция ползунка **корректно обновляются** (в т.ч. clamp `bonusToSpend`).
2. В сводке заказа отображаются **согласованные** суммы: субтотал после промо, списание бонусов (`promoPreview.bonusSpent`), товары к оплате, доставка, итог.
3. При оформлении заказа на сервер уходит то же значение, что видит пользователь (`bonusPointsToSpend` в теле заказа).
4. Нет заметных лагов из-за частых вызовов превью при движении ползунка (debounce/throttle или разделение «мгновенный UI» / «превью по отпусканию»).
5. Базовая **a11y** ползунка и читаемые подписи.

## Риски и примечания

- **Начисление бонусов** в превью завязано на `subtotalAfterPromo` и настройки `earn_percent` — при изменении списания текст «Начислим бонусов» может меняться; это ожидаемо, в UI можно кратко пояснить, что начисление считается от суммы после промо (как сейчас на сервере в `preview.post.ts`).
- Дублирование логики `max` на клиенте минимизировано: опора на `maxBonusForOrder` из превью + баланс в `maxBonusAvailable` уже в computed.
- Бонусами не должны оплачиваться сервисные позиции (в т.ч. доставка); для слайдера это означает, что "потолок" может быть заметно ниже простого `% от корзины`, и UI должен это нормально объяснять подписью.
- Требуется целочисленное поведение (`step=1`, без дробных бонусов) и визуально предсказуемое округление, чтобы не спорить с серверной/кассовой математикой.
- Для `internal`-режима сервер должен оставаться защищён от двойного списания (конкурентные заказы), а UI в ошибке показывать понятный фолбэк "бонусов недостаточно / лимит изменился" и предлагать пересчёт.
- Начисление — только по финальному статусу заказа; слайдер и блок превью не должны создавать ожидание "мгновенного" начисления в текущем заказе.
- Комбинация "промокод + бонусы" может быть запрещена настройкой ресторана: UI должен уметь корректно отразить такое ограничение (через `preview`/серверный флаг, без локальной эвристики).
