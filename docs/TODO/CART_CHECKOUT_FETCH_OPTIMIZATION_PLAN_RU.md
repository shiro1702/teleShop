# План: лишние запросы на страницах корзины и оформления заказа

## Связь с другими задачами

Запросы **`/api/tenant`** и общая модель загрузки профиля ресторана (тема, юрданные, часы) описаны отдельно и **реализуются в другой ветке работ**:

- [План оптимизации запросов тенанта](./TENANT_FETCH_OPTIMIZATION_PLAN_RU.md)

В **этом** документе не дублируются этапы из `useTenant`, кроме пересечений по точкам входа (`app.vue`). Здесь — всё остальное: лояльность, адреса, рестораны и зоны, промо-превью, доставка и навигация между `/cart` и `/checkout`.

---

## Контекст

Страницы **`/{city}/{tenant}/cart`** и **`/{city}/{tenant}/checkout`** оба монтируют один и тот же [`pages/checkout.vue`](../../pages/checkout.vue) через разные обёртки ([`pages/cart.vue`](../../pages/cart.vue), [`pages/[city_slug]/[tenant_slug]/checkout.vue`](../../pages/[city_slug]/[tenant_slug]/checkout.vue)). При SPA-переходе между ними выполняется **повторный mount** большого экрана оформления → снова срабатывают `onMounted`, watchers с `immediate`, повторный `loadRestaurants()` и т.д.

Плюс на **одном** монтировании возможны дубли из-за гидратации Supabase, нескольких watchers на одни и те же данные и параллельной загрузки зон для нескольких филиалов.

Источники логики (основные):

| Область | Файлы |
|--------|--------|
| Корзина / шаги / промо / лояльность | [`pages/checkout.vue`](../../pages/checkout.vue) |
| Адресная книга и `/api/customer/addresses` | [`composables/useCheckoutAddress.ts`](../../composables/useCheckoutAddress.ts) |
| Филиалы, `/api/restaurants`, `/api/restaurant-zones` | [`composables/useCheckoutTenantRestaurants.ts`](../../composables/useCheckoutTenantRestaurants.ts) |
| Тема и shop context (пересекается с отдельным планом) | [`composables/useTenant.ts`](../../composables/useTenant.ts), [`app.vue`](../../app.vue) |

---

## Цели

1. Сократить **повторные** запросы к одним и тем же эндпоинтам при переходе **корзина → оформление** и при повторном показе экрана в рамках одной сессии вкладки и одного `shop_id`.
2. Убрать или минимизировать **внутристраничные** дубли (`loyalty/balance`, `customer/addresses`, двойная загрузка зон, лишний второй `loadRestaurants`, повторный `delivery-resolve` без смены координат).
3. Не ломать сценарии: смена ресторана/города, Telegram / MAX initData, выход из аккаунта, пустая корзина.

## Этап A — навигация и жизненный цикл экрана (высокий эффект на флоу)

**Проблема:** два route-файла приводят к размонтажу `checkout.vue` при переходе `/cart` → `/checkout`.

**Варианты (выбрать один направление в реализации):**

1. **Один маршрут для обоих шагов** — например один путь с query `step` или один сегмент вроде `order`, внутри которого меняется только состояние шага без смены page-компонента. Требует аккуратной миграции ссылок (`tenantPath('/cart')` / `tenantPath('/checkout')`) и редиректов со старых URL при необходимости.
2. **Если URL оставляем как есть** — ввести **сессионный кэш данных чекаута** (например composable или Pinia store с ключом `shopId` + `userId`), чтобы второй mount не дергал сеть без необходимости (см. этап E).

**Критерий готовности:** при переходе корзина → оформление без смены магазина и пользователя не повторяется полный «первый заход» по списку API (или число запросов не хуже заранее зафиксированного бюджета в чеклисте).

## Этап B — лояльность: `/api/loyalty/balance`

**Проблема:** в [`pages/checkout.vue`](../../pages/checkout.vue) `watch([supabaseUser, shopIdFromRoute], …, { immediate: true })` даёт два срабатывания при появлении сессии после гидратации.

**Сделать:**

- Не вызывать `loadLoyaltyBalance()` до тех пор, пока не стабилизирована авторизация **или** дедуплицировать одинаковые подряд вызовы (например по ключу `userId|shopId` + флаг «уже загружено для этого ключа» с TTL или инвалидацией после заказа).
- Унифицировать заголовок **`x-shop-id`**: сейчас часть запросов идёт со slug из маршрута, часть — с UUID из `tenant.shopId`; для API заказов и лояльности зафиксировать один источник истины по правилам бэкенда.

**Критерий:** один реальный запрос баланса на монтирование при стабильной сессии (если не принудительное обновление).

---

## Этап C — адреса: `/api/customer/addresses`

**Проблема:** в [`useCheckoutAddress.ts`](../../composables/useCheckoutAddress.ts) watcher по `shopId`, `supabaseUser`, storage key с `immediate: true` вызывает `loadSavedAddresses()` несколько раз при появлении пользователя и после установки `tenant.shopId`.

**Сделать:**

- Аналогично лояльности: ждать стабильной пары **сессия + shop id**, либо дедуп in-flight запросов GET.
- После выполнения [плана по тенанту](./TENANT_FETCH_OPTIMIZATION_PLAN_RU.md) убедиться, что лишнее изменение `shopId` в состоянии не триггерит повторную полную загрузку адресной книги без причины.

**Критерий:** один GET адресной книги при первом готовом контексте (плюс повторы только после явных действий пользователя или инвалидации).

---

## Этап D — рестораны и зоны: `/api/restaurants`, `/api/restaurant-zones`

**Проблемы:**

1. В [`checkout.vue`](../../pages/checkout.vue) `loadRestaurants()` вызывается из `onMounted` и из `watch(shopIdFromRoute)` (и при появлении messenger `initData`) — возможны пересечения при одном и том же `shop_id`.
2. В [`useCheckoutTenantRestaurants.ts`](../../composables/useCheckoutTenantRestaurants.ts) **`loadAllRestaurantZones`** запрашивает зоны для **каждого** филиала с доставкой, а **`watch(selectedRestaurantId)`** с `immediate: true` **дополнительно** запрашивает зоны для выбранного филиала → дублирование хотя бы для одного `restaurant_id`.

**Сделать:**

- Гарантировать **один** вызов `loadRestaurants` на «холодный» вход при неизменном `shopId` (флаг инициализации или сравнение предыдущего ключа магазина).
- После загрузки списка ресторанов использовать **`allRestaurantZones`** для текущего `selectedRestaurantId` без второго запроса в watcher **или** не вызывать `loadAllRestaurantZones` для всех филиалов на первом экране — только для выбранного, остальные — по смене филиала / по требованию карты (продуктовое решение).
- Явно задокументировать стратегию: «все зоны для карты» vs «ленивая подгрузка».

**Критерий:** нет двух одинаковых GET к `/api/restaurant-zones` с теми же query при одном сценарии загрузки; число запросов к ресторанам предсказуемо.

## Этап E — кэш сессии витрины (если маршруты cart/checkout не объединяются)

**Идея:** хранить в памяти вкладки (или `useState` с ключом по scope) последний успешный результат:

- список ресторанов и при необходимости зоны (по политике из этапа D);
- опционально «холодный» снапшот для первого шага.

Инвалидация: смена `shop_id`, выход из аккаунта, явное событие после заказа.

**Критерий:** повторный mount при переходе `/cart` → `/checkout` не повторяет тяжёлые запросы, если данные ещё актуальны.

---

## Этап F — промо и доставка: `/api/promo/preview`, `/api/delivery-resolve`

**Проблемы:**

- `promo/preview`: debounced watcher по сигнатуре корзины — при быстрых изменениях зависимостей возможны лишние вызовы; после загрузки лояльности сигнатура может измениться ещё раз.
- `delivery-resolve`: несколько watchers (координаты сохранённого адреса, геокодинг) могут вызвать resolve подряд с теми же координатами.

**Сделать:**

- Для превью: убедиться в стабильности `promoPreviewWatchSignature`, не включать в неё поля, которые меняются после побочных загрузок без изменения корзины; при необходимости увеличить debounce или добавить сравнение «тело запроса не изменилось — не слать».
- Для delivery: кэш последнего успешного `(lat, lon, shopId)` → если совпадает, не вызывать `/api/delivery-resolve` повторно; инвалидировать при смене адреса или корзины, если бэкенд это требует.

**Критерий:** нет двух подряд идентичных POST к `delivery-resolve` без смены входных данных.
