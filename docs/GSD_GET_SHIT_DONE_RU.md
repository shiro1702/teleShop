# GSD (get-shit-done): практическая документация для TeleShop

Документ описывает, что такое GSD, как его применять в текущем проекте и какие команды использовать в ежедневной разработке.

---

## 1. Что это такое

`get-shit-done` (GSD) — это workflow-надстройка для AI-разработки:

- помогает не терять контекст на длинных задачах;
- делит большую задачу на фазы и атомарные планы;
- использует специализированных агентов для ресерча, планирования, реализации и проверки;
- сохраняет артефакты процесса в `.planning/*`;
- поддерживает аккуратную работу с git (маленькие понятные изменения, проверяемые результаты).

Проще говоря: GSD делает работу AI-помощника более предсказуемой и воспроизводимой.

Источник: [gsd-build/get-shit-done](https://github.com/gsd-build/get-shit-done)

---

## 2. Зачем использовать в TeleShop

Для этого репозитория GSD полезен, когда задача:

- затрагивает несколько слоев (`pages`, `server/api`, `middleware`, `supabase`);
- требует явных решений по архитектуре (тенанты, роли, платежи, биллинг);
- должна иметь проверяемый результат и прозрачную историю изменений.

GSD особенно полезен для фич вида:

- multi-tenant доступ и роли;
- платежный контур (YooKassa / Т-Банк);
- onboarding и админские flow в dashboard;
- большие рефакторинги middleware и API.

---

## 3. Базовый цикл работы

Рекомендуемый цикл по фазам:

1. `discuss-phase` — зафиксировать ожидания и спорные решения.
2. `plan-phase` — получить конкретные атомарные шаги.
3. `execute-phase` — реализовать шаги (в т.ч. параллельно, где возможно).
4. `verify-work` — пройти user acceptance проверку.
5. `ship` — подготовить PR с осмысленным описанием.

Это снижает риск "написали код, но не то, что имелось в виду".

---

## 4. Быстрый старт (минимум команд)

Если проект уже существует (как TeleShop):

1. Сначала карта кодовой базы:

```bash
/gsd:map-codebase
```

2. Затем полноценный запуск процесса:

```bash
/gsd:new-project
```

3. Для текущей фазы (пример: фаза 2):

```bash
/gsd:discuss-phase 2
/gsd:plan-phase 2
/gsd:execute-phase 2
/gsd:verify-work 2
```

4. Подготовка поставки:

```bash
/gsd:ship 2
```

## 5. Когда использовать quick/fast режимы

### 5.1. `quick`

Подходит для средних ad-hoc задач без полного фазового цикла:

```bash
/gsd:quick
```

Полезно, если нужно быстро добавить небольшую фичу, но с сохранением структуры и артефактов.

### 5.2. `fast`

Для совсем небольших правок без планирования:

```bash
/gsd:fast <описание-задачи>
```

Использовать, когда изменение локальное и не затрагивает архитектурные решения.

## 6. Что создается в проекте

Типовые артефакты:

- `PROJECT.md` — общее видение проекта;
- `REQUIREMENTS.md` — требования и границы scope;
- `ROADMAP.md` — фазовый план;
- `STATE.md` — текущее состояние/решения;
- `.planning/*` — файлы исследования, планов, суммаризации и верификации.

Практический эффект: любая новая сессия быстрее "вкатывается" в контекст и меньше ломает ранее принятые решения.

---

## 7. Рекомендации по применению в команде

- Не пропускать `discuss-phase` для спорных UX/архитектурных задач.
- В `plan-phase` проверять, что планы атомарные и проверяемые.
- После `execute-phase` обязательно проходить `verify-work`, а не только смотреть на "код скомпилился".
- Для PR использовать `ship`, чтобы автоматически получить понятный summary и test plan.
- Для "внезапных" задач использовать `quick`, чтобы не ломать основной roadmap.

---

## 8. Ограничения и здравый смысл

GSD ускоряет работу, но не отменяет инженерные решения:

- доменные решения по платежам, безопасности и доступам должны быть явно подтверждены человеком;
- критичные изменения в продовых потоках нужно проверять вручную (UAT + smoke);
- если задача маленькая, лучше `fast`/обычная правка, чем перегруженный процесс.

---

## 9. Краткий чеклист перед запуском фазы

- Цель фазы сформулирована в 1-2 предложениях.
- Понятны ограничения (безопасность, роли, tenant boundaries, платежные инварианты).
- Есть критерии "готово" (что пользователь должен уметь сделать после фазы).
- Выбран режим: `phase` цикл или `quick`.
- Определено, как будет проходить проверка (`verify-work`).
