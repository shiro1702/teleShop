# TeleShop: мультитенантная SaaS-платформа для ресторанов

Документ описывает целевую архитектуру, доменную модель, роли, структуру дашборда и функционал для превращения текущего проекта (Nuxt 3, Supabase, Telegram) в мультитенантную платформу с несколькими городами, ресторанами, филиалами и гибким меню.

Единообразие терминов зафиксировано в `docs/TERMS.md`:
- внутренний дашборд: «Филиал»;
- клиентская витрина: «Ресторан».

## Фундамент мульти-города без лишнего интерфейса

Цель: заложить **мульти-город в данных и URL**, но **не тратить время** на попап «Выберите город», геолокацию и сложный city-picker. Пока в системе один город, пользователь живёт как на обычном сайте: заходит на домен — его незаметно ведут на единственный slug.

### Шаг 1: База данных (Supabase)

1. Создай таблицу **`cities`** (минимум):

   - `id` — `uuid` primary key default `gen_random_uuid()` **или** целочисленный `serial`/`bigint` — как принято в вашем проекте; для Supabase чаще удобен **uuid**.
   - `name` — текст, например `Улан-Удэ`.
   - `slug` — текст, **уникальный**, например `ulan-ude`.
   - По желанию: `is_active`, `sort_order`, `created_at`.

2. Добавь **ровно одну** строку:

   | Поле | Значение |
   |------|----------|
   | `name` | Улан-Удэ |
   | `slug` | `ulan-ude` |

   Если используете uuid — зафиксируйте этот `id` в seed-миграции или в `.env` / `runtimeConfig` как **дефолтный город**, чтобы в коде создавать сущности с известным `city_id`.

3. Во **все основные таблицы**, которые привязаны к географии или к витрине города, добавь колонку **`city_id`** с foreign key на `cities(id)`:

   - `restaurants` (или ваша таблица филиалов точек — как она названа в БД)
   - `users_addresses` (адреса клиентов)
   - `orders`

   При **создании** ресторана, адреса и заказа всегда проставляй `city_id` этого единственного города (из конфига или константы), пока не появится второй город.

4. Постепенно фильтруй API-запросы по `city_id` (или по `slug` города из URL), чтобы при добавлении второго города не переписывать всё с нуля.

### Шаг 2: Структура ссылок (Nuxt 3)

Публичная витрина с первого дня должна жить **в подпапке с slug города**:

- `https://chopoest.ru/ulan-ude` — главная (каталог / лендинг города)
- `https://chopoest.ru/ulan-ude/:restaurant-slug/cart` — корзина конкретного ресторана
- `https://chopoest.ru/ulan-ude/:restaurant-slug/checkout` — оформление конкретного ресторана
- Дальше по той же схеме: все маршруты витрины под `pages/[city_slug]/...`

Дашборд, логин и служебные страницы можно оставить **без** префикса города (например `/login`, `/dashboard`), если они не привязаны к витрине.

Имена файлов в `pages/`: динамический сегмент `[city_slug]` совпадает с полем `slug` в таблице `cities`.

### Шаг 3: «Магия» редиректа (Nuxt middleware)

Пока город один, пользователю **не нужно** вручную вводить `/ulan-ude/` в адресной строке.

1. Добавь **глобальный** middleware, например `middleware/redirect-city.global.ts`.

2. Логика (минимум):

   - Если пользователь зашёл на **корень сайта** — путь ровно `/` (и при необходимости вариант с trailing slash, если у вас он включён) —
   - выполни **немедленный** `navigateTo('/ulan-ude')` (или к `/${defaultCitySlug}`).

3. **Не** редиректить с путей вроде `/login`, `/api`, статики и т.д. — только с истинного корня витрины.

4. Чтобы потом не искать по коду, вынеси slug дефолтного города в **`runtimeConfig.public.defaultCitySlug`** (значение `ulan-ude`) и используй его в middleware и при генерации ссылок в шаблонах.

Пример middleware (адаптируй под `runtimeConfig` и правила trailing slash):

```ts
// middleware/redirect-city.global.ts
export default defineNuxtRouteMiddleware((to) => {
  const config = useRuntimeConfig()
  const slug = config.public.defaultCitySlug as string
  if (!slug) return
  if (to.path === '/' || to.path === '') {
    return navigateTo(`/${slug}`)
  }
})
```

Итог: в адресной строке всегда канонический URL с городом; с точки зрения UX человек просто открыл сайт — редирект короткий и незаметный.

## 1. Терминология

| Термин | Смысл |
|--------|--------|
| **Платформа** | Ваш SaaS: единое приложение, биллинг, общий бот опционально. |
| **Тенант (организация)** | Юридически/коммерчески одна компания-оператор. В коде это сущность `shop`. Организация = Бренд. |
| **Город** | Географическая/операционная единица платформы. В каждом городе может быть несколько организаций. |
| **Филиал** | Точка продаж конкретной организации: адрес, часы, зоны доставки, локальные цены и стоп-листы. Соответствует сущности `restaurant`. |
| **Меню** | Категории, позиции, **параметры** (вариант позиции: размер/вес/объём/количество и т.п.) и **модификаторы** (добавки, соусы и т.п.); глобально привязано к организации (`shop_id`), но может переопределяться на уровне филиала (`restaurant_id`). |

## 2. Иерархия и мультитенантность

### 2.1. Логическая иерархия

```
Платформа
  └── Организация (Shop / Бренд)
        ├── Город 1
        │    └── Филиалы (Restaurants)
        └── Город 2
             └── Филиалы (Restaurants)
```

Практический вывод для модели данных:

- Связь: `organization (shop)` -> `branch (restaurant)`.
- Каждый филиал привязан к конкретному городу (`city_id`).
- Это позволяет одной организации работать в нескольких городах через свои филиалы.

### 2.2. Изоляция данных

- Каждый запрос к API и к БД должен быть ограничен **`organization_id` / `shop_id`** (как у вас сейчас для `shop`).
- Row Level Security в Supabase: политики вида `organization_id = auth.jwt() ->> 'org_id'` (или через membership-таблицу).
- Публичная витрина: резолвинг по **`slug`**, кастомному домену или deep-link из Telegram (`start_param` с привязкой к филиалу/ресторану).

### 2.3. Меню: уровни

1. **Меню организации** — базовый каталог для всей сети (`shop_id`).
2. **Меню филиала** — модель **наследование + исключения** (скрыть позицию, другая цена, другой стоп-лист).

Рекомендация по схеме: таблицы `products` (глобальное меню), и отдельная таблица `restaurant_product_overrides` (`restaurant_id`, `product_id`, `price_override`, `is_hidden`, `is_in_stop_list`) для переопределений на уровне филиала.

### 2.4. Меню: параметры позиции и модификаторы (разные сущности)

- **Параметры** описывают **вариант самой позиции**: один измеримый аспект на группу (например только «диаметр» или только «объём» или только «количество штук»). У каждого варианта — **своя цена** (база для строки заказа) и опционально **атрибуты без влияния на цену** (например вес порции, тип теста, если для тонкого/традиционного цена одинаковая). Набор доступных вариантов **задаётся на уровне конкретного товара** (у одной пиццы есть 25/30/35 см, у другой — только 30/35 см).
- **Модификаторы** — **надстройка** к уже выбранному варианту: добавки, соусы, «убрать лук», режимы ценообразования «+сумма» или «×множитель» к цене после параметров. Могут наследоваться с категории с возможностью отключения на товаре (см. `pages/dashboard/menu.md`).

Подробнее по UX, примерам и целевой схеме БД — в `pages/dashboard/menu.md`; отображение в корзине — в `pages/[city_slug]/[tenant_slug]/cart.md`.

## 3. Роли пользователей

### 3.1. Базовые роли

| Роль | Описание |
|------|----------|
| **Владелец ресторана (Owner)** | Полный доступ: биллинг, удаление организации, все филиалы, все менеджеры, интеграции, боты. |
| **Менеджер ресторана (Manager)** | Ограничиваемый набор прав (см. ниже). |

### 3.2. Права менеджера (гранулярность)

Примеры флагов/permission-записей:

- Просмотр / редактирование **заказов**
- Управление **филиалами** (все или только назначенные)
- Управление **меню** (только чтение / полное редактирование)
- Управление **зонами доставки**
- Управление **акциями и промокодами**
- Доступ к **аналитике и отчётам**
- Приглашение других менеджеров (опционально только для Owner)

Хранение: таблица `organization_members` (`user_id`, `organization_id`, `role`, `permissions_json` или нормализованная `member_permissions`).

### 3.3. Пользователь конца (клиент)

- Регистрация/вход через Telegram (как сейчас) и/или email при необходимости.
- Привязка к платформенному боту или боту ресторана — см. раздел 4.

## 4. Telegram-боты

См. детальный план омниканальной мультитенантной маршрутизации и интеграции MAX:
- `docs/OMNICHANNEL_MULTITENANT_PLAN_RU.md`

### 4.1. Собственный бот ресторана

- В настройках организации: `telegram_bot_token`, `telegram_bot_id`, webhook URL на ваш бэкенд с идентификацией организации (секрет пути или verified token → `shop_id` / `org_id`).
- Уведомления менеджерам: `manager_chat_id` или несколько чатов по филиалам.

### 4.2. Платформенный бот

- Один бот для онбординга: пользователь стартует с `start` payload, указывающим ресторан/филиал.
- Регистрация клиента в БД с привязкой к `organization_id` и опционально `branch_id`.
- Ограничение: в одном чате с платформенным ботом нужно однозначно понимать контекст заказа (последний выбранный филиал или явный выбор в UI Mini App).

### 4.3. Общие требования

- Валидация `initData` на сервере (уже в духе текущего кода).
- Маппинг `bot_id` → организация для мультисценария.

### 4.4. Подключение webhook для бота ресторана из сайта

Ниже целевой сценарий, чтобы владелец ресторана сам подключал Telegram-бота через дашборд без ручной работы разработчика.

1. **Где в интерфейсе**
   - Раздел: `/dashboard/settings/organization` -> блок `Telegram бот`.
   - Поля:
     - `Bot Token` (вводится вручную владельцем);
     - переключатель `Режим: свой бот / платформенный бот`;
     - кнопка `Подключить бота`.

2. **Что происходит после нажатия `Подключить бота`**
   - фронт отправляет токен на серверный endpoint (например, `POST /api/dashboard/integrations/telegram/connect`);
   - сервер вызывает `getMe` у Telegram API и проверяет, что токен валиден;
   - сервер формирует webhook URL вида:
     - `https://your-domain.ru/api/webhook/telegram/<organization_id>/<secret>`
     - либо единый URL + `secret_token`, если используете заголовок `X-Telegram-Bot-Api-Secret-Token`;
   - сервер вызывает `setWebhook` в Telegram API;
   - при успехе сохраняет в БД:
     - `telegram_bot_token` (в зашифрованном виде),
     - `telegram_bot_id`,
     - `telegram_bot_username`,
     - `webhook_secret`,
     - `webhook_status=active`,
     - `webhook_last_checked_at`.

3. **Что показать пользователю в UI**
   - статус: `Подключен` / `Ошибка` / `Отключен`;
   - username бота (`@name`) и время последней проверки;
   - кнопки:
     - `Переподключить webhook`;
     - `Отключить webhook` (внутри вызывает `deleteWebhook`).

4. **Безопасность (обязательно)**
   - токен бота не отдавать на клиент после сохранения;
   - хранить токен шифрованно (или в secret storage);
   - webhook endpoint валидирует источник:
     - по `secret_token`/секрету в URL,
     - по соответствию `bot_id` и `organization_id`;
   - доступ к подключению/отключению только у роли `Owner` (или менеджер с отдельным permission).

5. **Операционные проверки**
   - endpoint `Проверить webhook` в дашборде (пинг + `getWebhookInfo`);
   - если Telegram возвращает ошибку (`401`, `wrong token`, `webhook is not set`) — показать понятный текст и action «Исправить»;
   - логировать события подключения: кто подключил, когда, для какой организации.

6. **Минимальный UX-текст в интерфейсе**
   - «Подключая своего бота, вы разрешаете платформе получать обновления Telegram и обрабатывать сообщения для вашего ресторана».
   - «Доставку и обработку заказов выполняет ваш ресторан; платформа предоставляет технологическую витрину и инструменты».

## 5. Функционал (чеклист и дополнения)

- **Заказы**: статусы, кухня/курьер, история, печать чека, экспорт

- **Экран повара (KDS)**: См. `pages/dashboard/kitchen.md` для стратегии отображения заказов на кухне (маршрутизация по цехам, группировка позиций).
- **Audit log (MVP)**: фиксируем как минимум
  - смены статусов заказа (с причиной/комментариями менеджера);
  - операционные изменения: меню, филиалы, зоны доставки, настройки интеграций/Telegram webhook.
- **Оплаты**: наличные, карта, интеграция эквайринга; возвраты
- **POS и фискализация**: интеграция с **iiko** и онлайн-кассой (54-ФЗ), статусы чека, возвраты чеков
- **Уведомления**: push/Telegram/SMS по событиям заказа
- **Заявки / лидогенерация**: форма «стать партнёром», модерация админом платформы
- **Админка платформы** (отдельно от дашборда ресторана): города, модерация ресторанов, каталог модулей и их лимитов
- **Биллинг и подписки**: оплата модулей (feature toggles), usage-лимиты, отключение за неоплату
- **Аудит**: кто изменил меню/цены/зоны
- **Импорт/экспорт** меню (CSV/Excel)
- **SEO и публичные страницы** по городу/ресторану (если не только Mini App)
- **API и вебхуки** для интеграций (POS, агрегаторы)
- **Резервы и столики** (если актуально позже)

## 6. Публичная витрина агрегатора (главная и служебные страницы)

Важно зафиксировать это в продукте и в тексте на сайте с первого дня:

- TeleShop/Chopoest работает как **агрегатор ресторанов**.
- **Доставку выполняет каждый ресторан самостоятельно** (свои курьеры/логистика и свои зоны).
- Платформа даёт витрину, заказы, интерфейс и инструменты управления.

### 6.1. Сценарии главной страницы города `/:city_slug`

Главная страница должна поддерживать минимум такие пользовательские потоки:

1. **Выбрать ресторан из списка**:
   - карточки ресторанов;
   - быстрые фильтры (доставка/самовывоз, «открыто сейчас», категория кухни).

2. **Искать еду по названию** (глобально по городу):
   - поиск по товарам меню;
   - в результате показывать позицию + ресторан + филиал/зону, где доступно.

3. **Искать ресторан по названию**:
   - отдельная вкладка или единый поиск с типами результатов (блюда/рестораны).

4. **Открыть карточку ресторана** и перейти к заказу:
   - меню, условия доставки, минималка, время работы, контакты.

5. **Проверить адрес доставки**:
   - пользователь вводит адрес, система показывает, какие рестораны/филиалы доставляют в эту точку.

### 6.2. Страница для партнёров (B2B)

Нужна отдельная публичная страница для привлечения ресторанов, например:

- `/partners`

Состав страницы:

- кто вы и как работаете;
- что получает ресторан (бот, сайт, заказы, аналитика, менеджеры, меню);
- тарифы/прайс и модель оплаты;
- как проходит подключение (шаги онбординга);
- FAQ для владельцев ресторанов;
- CTA-кнопка: **«Подключить ресторан»**.

Целевое действие:

- переход с `/partners` в дашборд/онбординг, например на `/register` или `/onboarding`, где партнёр создаёт ресторан.

#### Прайс для партнёров (модульная модель)

Валюта: RUB, оплата по модулям (чаще ежемесячно, для отдельных модулей возможны weekly/usage-пакеты).

Актуальный каталог модулей, цен и зависимостей:

- `docs/FEATURE_TOGGLES_PRICING_RU.md`

Коротко по принципу:

- базовое ядро (QR-меню + Telegram-поток) может быть бесплатным как low-friction вход;
- операционные и маркетинговые возможности подключаются отдельными тумблерами;
- white-label, доставка, продвижение и интеграции продаются как отдельные premium-модули;
- при необходимости sales может предлагать "пакеты", но это пресеты тумблеров, а не жесткие планы.

Разовые платежи:

- Онбординг и настройка интеграций: от **15 000 ₽** (по объёму).
- Миграция меню/данных: от **10 000 ₽**.

Логика лимитов:

- Лимиты проверяются по активным брендам/филиалам и по usage отдельных модулей (если модуль usage-based).
- Определение “активного филиала” (для лимитов модулей):
  - филиал считается активным, если за последние 30 дней есть хотя бы 1 успешный реальный заказ, начиная со статуса `принят в работу`;
  - `отменен` не учитывается как успешный.
- Определение “активного бренда”:
  - бренд активен, если у него есть хотя бы 1 активный филиал (по правилам выше).
- При превышении лимита пользователь может включить/расширить соответствующий модуль. Текущая работа не блокируется мгновенно: сначала предупреждение и CTA на апгрейд.

#### Условия пользования платформой для партнёров

1. **Модель сервиса**
   - Платформа предоставляет сайт, инструменты управления, приём и маршрутизацию заказов.
   - Ресторан самостоятельно выполняет приготовление, доставку, коммуникацию по качеству и соблюдение сроков.

2. **Финансовые условия**
   - Абонплата списывается ежемесячно.
   - Комиссия с заказов на текущем этапе не взимается.

3. **Роли и доступ**
   - Ответственность за доступы сотрудников несёт владелец организации.
   - Партнёр обязан своевременно удалять доступ у уволенных сотрудников.

4. **Контент и правомерность**
   - Ресторан отвечает за корректность меню, цен, фото и юридически значимой информации.
   - Запрещён контент, нарушающий законы РФ и права третьих лиц.

5. **Интеграции и Telegram**
   - При подключении собственного бота ресторан подтверждает право владения токеном.
   - Партнёр обязан не передавать токены третьим лицам и соблюдать требования безопасности.

6. **SLA и поддержка**
   - Базовая поддержка включена в ядро.
   - Расширенный SLA и выделенный канал поддержки могут быть отдельным premium-модулем или enterprise-пакетом продаж.

7. **Приостановка и расторжение**
   - При неоплате: сначала блок дашборда (доступ к управлению отключается), затем заморозка аккаунта/доступа после 3 месяцев неуплаты.
   - Предупреждение о риске заморозки — заранее (срок определяется регламентом биллинга).
   - При расторжении предоставляется экспорт данных заказов и меню (в пределах возможностей API/форматов).

8. **White-label политика**
   - По умолчанию допускается присутствие агрегатора по правилам UI.
   - При включенном модуле `white_label` упоминания агрегатора убираются из основной навигации, остаются только юридические ссылки в футере.

9. **iiko и онлайн-касса**
   - Партнёр самостоятельно предоставляет действующие доступы/ключи к iiko и провайдеру онлайн-кассы.
   - Платформа передаёт заказ в iiko и инициирует фискализацию по согласованному сценарию (предчек/чек оплаты).
   - Юридическая ответственность за корректность фискальных данных, налоговых ставок, номенклатуры и соответствие 54-ФЗ остаётся на партнёре.
   - При недоступности внешнего провайдера (iiko/касса) платформа логирует ошибку и показывает статус, но не гарантирует мгновенное восстановление на стороне провайдера.

### 6.3. Политика шапки сайта (агрегатор vs ресторан)

Принятое решение: поведение шапки зависит от контекста URL и включенных модулей ресторана.

1. **Режим по контексту URL**
   - `/:city_slug` -> агрегаторный режим (поиск и выбор ресторана).
   - `/:city_slug/:tenant_slug` -> ресторанный режим (брендовая витрина ресторана).

2. **Модель доступа (фиксируем в MVP)**
   - По умолчанию: сайт ресторана на платформе, с мягкой интеграцией агрегатора.
   - При включенном `white_label`: full white-label.
   - Кастомный домен может быть условием модуля `white_label` или отдельным модулем.

3. **Правило ссылки «Все рестораны»**
   - Показывается только на главной ресторана `/:city_slug/:tenant_slug`.
   - В `cart` и `checkout` не показывается, чтобы не уводить из оформления.
   - Если `white_label` не включен, ссылка ведёт на город текущего ресторана: `/${defaultCitySlug}` из настроек ресторана (или его фактический `city_slug`).

4. **Кастомный домен**
   - Доступность задается модулем (`white_label` или отдельный `custom_domain`).
   - На кастомном домене шапка и навигация полностью брендовые.

4.1. **Если бренд временно не работает**
   - Для платформенного домена: редирект на страницу города (выбранный `defaultCitySlug`).
   - Для бренда с `white_label` на кастомном домене: редирект на страницу города в рамках того же кастомного домена (город доступен через его `city_slug`).

5. **White-label поведение**
   - Без `white_label`: допускается ненавязчивое присутствие агрегатора (ссылка «Все рестораны» на главной ресторана).
   - С `white_label`: убираем упоминания агрегатора из шапки и основного UI.
   - С `white_label`: оставляем только юридическую ссылку в футере (оферта/политика/контакты).

6. **Зачем это нужно продукту**
   - Рестораны получают изолированный брендовый сайт.
   - Платформа продолжает продвигать агрегатор там, где `white_label` не включен.
   - Переходы между режимами прозрачны и не ломают пользовательский поток заказа.

## 10. Краткие технические заметки

- **Nuxt**: разделить публичные страницы и дашборд (layout’ы `default` vs `dashboard`).
- **Supabase**: ключевые таблицы должны включать `shop_id`; для витрины и операционной логики также нужны `city_id` там, где это влияет на доступ/выборку. Политики RLS тестировать отдельно для Owner и Manager.
- **Секреты**: токены ботов хранить зашифрованно или в секрет-хранилище; в клиент не отдавать.
- **Секреты интеграций (MVP)**: токены iiko/онлайн-кассы хранить только на сервере (server-only columns) и не отдавать в клиент. В MVP допускается хранение в plaintext внутри server-only (без публичной утечки); шифрование можно добавить в следующих релизах.
- **Consent и ретеншн (MVP, минимально)**:
  - хранить факт согласия (оферта/политика) и timestamp минимально (например, при регистрации/первом входе);
  - заказы хранить до удаления аккаунта/в пределах требований закона, без жесткого SLA “в днях” на старте (уточнить позже).
- **Стратегия миграций БД (рекомендуемая)**:
  - `add nullable column` -> `backfill` (через migration/job) -> проверка -> `add indexes` -> затем по релизам сделать `not null` и/или переключить API с legacy-логики на новую.

4. `shop_members`
   - добавить `permissions jsonb default '{}'::jsonb` для гранулярных прав менеджера;
   - `restaurant_id` оставить как optional scope для филиала;
   - рассмотреть расширение до `city_id` (если нужно ограничивать доступ менеджера по городу).

5. `restaurant_delivery_zones`
   - прямой `city_id` не обязателен, если `restaurant_id` уже ссылается на филиал с `city_id`;
   - полезно добавить `updated_at` + индекс `(restaurant_id, is_active)`.

### 13.3. Важные ограничения и проверки

- Консистентность `shop_id`:
  - `orders.shop_id` должен совпадать с `restaurants.shop_id` для выбранного `restaurant_id`;
- Консистентность города:
  - `orders.city_id` должен совпадать с `restaurants.city_id`, если заказ на филиал.
- Для `cities` использовать `slug` как человекочитаемый ключ для URL, но в связях хранить `uuid`.

### 13.4. План безопасной миграции (без поломки текущего API)

1. Создать `cities`.
2. Добавить nullable-поля `city_id` в существующие таблицы.
3. Выполнить backfill:
   - `cities`: создать `ulan-ude`;
   - заполнить `restaurants.city_id`, `orders.city_id`;
4. Добавить NOT NULL и индексы после проверки данных.
5. Перевести API и UI на чтение/запись `city_id`.
6. Оставить legacy-маршруты и legacy-поля на переходный период, затем удалить.

### 13.5. Минимум для вашего ближайшего этапа

Если делать только необходимое сейчас:

- обязательно: `cities`, `restaurants.city_id`, `orders.city_id`;
- можно отложить: глубокие permissions, полная нормализация меню по филиалам.

## 14. Стратегия: QR-меню и онлайн-оплата

В рамках развития платформы (после MVP) необходимо реализовать сценарии для гостей, находящихся непосредственно в заведении (dine-in), а также полноценную онлайн-оплату.

### 14.1. QR-меню (Dine-in и Takeaway)

Развитие функционала QR-меню должно идти от простого к сложному, чтобы быстро дать ценность ресторанам и фудтракам.

**Сценарий:**
1. Гость сканирует QR-код (на столике, на кассе фудтрака, на тейбл-тенте).
2. Открывается веб-витрина бренда в контексте конкретного филиала.
   - URL: `https://domain.com/:city_slug/:tenant_slug/qr?branch_id=<UUID>&table=5`
     - `branch_id` (UUID) — **филиал/restaurant_id**, от которого нужно показать меню и на который должен “проваливаться” заказ.
     - `restaurant_id` — поддерживается как алиас для `branch_id` (на случай несовпадения имен параметров в макетах/генераторе QR).
     - `table` опционален (для сценария “заказ за стол” в будущем/следующих этапах).
3. Гость видит актуальное меню (с учетом стоп-листов именно этого филиала).

**Этапы реализации (от простого к сложному):**

**Этап 1: Только просмотр (Электронное меню)**
- Гость просто смотрит меню с актуальными ценами и фото.
- Заказ принимает официант или кассир голосом.
- *Ценность:* Экономия на печати бумажных меню, всегда актуальные цены и стоп-листы.

**Этап 2: Заказ без кассира (Очередь / Фудтраки / Фастфуд)**
- Идеально для фудтраков, кофеен to-go и заведений без официантов.
- Гость сканирует QR-код возле кассы, собирает корзину, оплачивает онлайн.
- Заказ падает на кухню (KDS/Telegram) с номером заказа (например, "Заказ №42").
- Гость ждет, когда назовут его номер или он появится на экране готовности.
- *Ценность:* Разгрузка кассира, устранение очередей, увеличение пропускной способности.

**Этап 3: Полноценный Dine-in с разветвлением (За стол или С собой)**
- При сканировании QR-кода гостю предлагается выбор формата:
  - **Заказ за стол:** Если в QR-коде зашит номер стола (или гость вводит его сам), заказ сразу уходит на кухню с пометкой "Стол 5". Официант только выносит блюда.
  - **Заказ с собой (Takeaway):** Гость сидит в заведении, но заказывает еду навынос. Заказ падает на кухню с пометкой "С собой", упаковывается в контейнеры.
- В корзине при таких сценариях скрываются поля адреса доставки.

**Техническая реализация:**
- В дашборде филиала (`/dashboard/branches/:id/qr`) генератор QR-кодов.
- Настройка типов QR-кодов: "Общее меню", "Для стола №X", "Для кассы (без стола)".
- Возможность скачать PDF с готовыми макетами для печати.
- Прокидывание контекста филиала в оформление:
  - фронтенд при переходе “в корзину” сохраняет `branch_id` (или алиас `restaurant_id`) в query параметрах `/cart`;
  - страница `/cart`/`/checkout` автоматически подставляет `selectedRestaurantId` из `branch_id`, чтобы гостю **не нужно было вручную выбирать филиал/адрес** на этапе корзины;
  - на сервере `restaurantId` дополнительно валидируется относительно текущего `shop_id` (tenant-уровень), чтобы исключить подмену филиала из URL.

### 14.2. Онлайн-оплата

Детальная архитектура и потоки вынесены в:

- `docs/PAYMENTS_RU_YOOKASSA_TBANK.md` — единый документ по YooKassa/Т-Банк, webhooks, статусам, возвратам и модели хранения ключей.
- `docs/SAAS_BILLING_RU.md` — биллинг подписки платформы (B2B).

Ключевой принцип для агрегатора:

- B2C-заказы гостей оплачиваются в merchant-контуре ресторана (деньги идут напрямую ресторану).
- B2B-абонплата за TeleShop идет через отдельный merchant-контур платформы.

Модель хранения ключей для B2C:

- MVP: ключи на уровне `shop` (один merchant на бренд/организацию).
- Эволюция: branch override на уровне `restaurant` (для сетей с разными договорами по филиалам).

По чекам и 54-ФЗ:

- при прямом эквайринге фискализация на стороне выбранного платежного провайдера/кассового контура ресторана;
- платформа передает корректный состав корзины и фиксирует статусы оплаты через webhook-механизм.

---

*Принятая целевая модель: одна организация (бренд) может работать в нескольких городах через свои филиалы.*

## 15. Сториз, Маркетинг и Лояльность
Платформа включает в себя систему для вовлечения пользователей, проведения акций и удержания клиентов.

**Документация разделена на следующие части:**
- **Сториз (Управление):** Как рестораны создают кампании, настраивают таргетинг, интерактивы (опросы, геймификация) — описано в `pages/dashboard/stories.md`.
- **Сториз (Отображение на витрине):** Как сториз персонализируются для клиентов, внедряются в каталог (Grid Injection) и обрабатывают диплинки — описано в `pages/[city_slug]/[tenant_slug]/index.md`.
- **Промокоды и бонусы (MVP):** CRUD промокодов в дашборде (`/dashboard/marketing`), кошелёк **1 бонус = 1 ₽**, начисление после успешной онлайн-оплаты, списание при оформлении заказа — подробно в `pages/dashboard/marketing.md`. Полноценный Rule Engine (счастливые часы, комбинирование акций), **«магазин за баллы»** и **грейды лояльности** в текущем этапе не реализуются; остаются целевой моделью для следующих релизов.
- **Продвинутый каталог:** Модификаторы, сборные комбо и кросс-селл (Upsell) в корзине — описано в `pages/dashboard/menu.md`.
